## HOW-facts (real IR/asm)

### Fact A — the entire post-opt LLVM-IR delta is a single `noalias` token
Full `diff` of `/tmp/s18-{off,on}/gmm.llir` (967 lines each), ignoring ModuleID/source_filename:
```
7c7
< define amdgpu_kernel void @gmm_kernel(... ptr addrspace(1)         nofree readonly captures(none) %2, ...)
> define amdgpu_kernel void @gmm_kernel(... ptr addrspace(1) noalias nofree readonly captures(none) %2, ...)
```
Zero instruction-level difference. `readonly captures(none)` is inferred on %2 in BOTH
(load-only use pattern), exactly as S8 Fact A; `noalias` adds only the alias promise.

### Fact B — the amdgcn INSTRUCTION STREAM is byte-identical (same md5)
Instruction lines only (`\t(s_|v_|ds_|global_|buffer_|scratch_|flat_|image_)…`), 968 lines each:
```
md5(off.ins) = a45bf8925aca27e7174969c16ee51ded
md5(on.ins)  = a45bf8925aca27e7174969c16ee51ded   ->  IDENTICAL
```
The ONLY amdgcn diff is a metadata-YAML line (`+ .actual_access: read_only` on the noalias
arg) — a descriptor annotation that changes no emitted instruction. Whole-kernel histogram is
identical ON vs OFF: `buffer_load 2, ds_ 32, global_load 8, s_load 3, v_readfirstlane 1`.

### Fact C — the group_sizes reload is a buffer-intrinsic, and it is NOT hoisted (either variant)
In the emitted LLIR the group_sizes read is `llvm.amdgcn.raw.ptr.buffer.load.i32` (the S6
buffer-ops path — TTGIR `amdg.buffer_load`), NOT a plain `load`/`global_load`:
```
entry  (blk 12) : %25  = tail call i32 @llvm.amdgcn.raw.ptr.buffer.load.i32(ptr addrspace(8) %22, i32 %24, 0, 0)   ; from _total_gmm_tiles
loop hdr (blk 117): %119 = tail call i32 @llvm.amdgcn.raw.ptr.buffer.load.i32(ptr addrspace(8) %22, i32 %24, 0, 0)   ; from _resolve_gmm_tile, INSIDE the loop
```
`%22 = make.buffer.rsrc.p8.p1(ptr addrspace(1) %2, …)` and `%24` (offset) are both defined in
the entry block ⇒ loop-invariant; the loop load %119 has *identical arguments* to the entry
load %25. The loop CFG is `117 (header) → … → 696 (atomic) → 698 (latch) → 117`, with
`%697 = atomicrmw add ptr addrspace(1) %5, i32 1 …` (tile_counter) at line 766 inside the loop.
Despite the reload being a textbook loop-invariant duplicate, it stays in the loop in BOTH:
```
                       raw.ptr.buffer.load.i32 count (module has 3: entry gs, loop gs, store rsrc build)
opt -passes=gvn                : off 3 / on 3   (no CSE of %25↔%119)
opt loop-mssa(licm)+gvn        : off 3 / on 3   (no hoist)
opt -O3 (fresh)                : off 3 / on 3   (no hoist, and -O3 diff = only the define token)
```

### Fact D — the atomic is NOT the barrier (S14 hypothesis refuted)
S14 hypothesized the in-loop `atomic_add(tile_counter_ptr)` may-aliases group_sizes and blocks
the hoist. Counter-test: patch the OFF module replacing the `atomicrmw` with a benign
`add i32 0, 1` (removes the write AND the sync/ordering barrier), re-run -O3:
```
no-atomic variant -O3 : raw.ptr.buffer.load.i32 = 3   (baseline OFF = 3)  -> STILL not hoisted
```
So the reload persists even with the atomic gone. The buffer-load intrinsic is simply not
GVN-CSE'd / LICM-hoisted across the backedge here (intrinsic-call value-numbering / MemorySSA
limitation on the buffer intrinsic) — there was never an alias edge for `noalias` to remove.

### Fact E — AA already proves NoAlias without the attribute; noalias adds no AA result
`opt -O3 -mcpu=gfx1250 -stats` line-diff OFF vs ON = EMPTY. Identical:
`aa 28 MayAlias / 10 MustAlias / 38 NoAlias` (unchanged), `gvn 34 deleted / 34 simplified`,
identical licm. The buffer-load's memory effect is `memory(argmem: read)` on the addrspace(8)
resource `%22`; the atomic writes addrspace(1) `%5`. AA sees these NoAlias by construction
(different address spaces + distinct provenance chains) — the addrspace(1) `noalias` on %2 is
redundant to that reasoning. This is the S8 result reproduced for gmm: the interfering write is
already NoAlias to the read, so the alias-gated middle-end opts have no firing site.

### Fact F — S3 soundness applied: group_sizes is never written

`out_ptr` (214, 647, 650, 812, 815), `tile_counter_ptr` (390), `bias_grad_ptr` (657, 822).

also passes it as a tensor distinct from out/tile_counter/bias_grad, satisfying the noalias
axis). The annotation is safe — it simply buys nothing.
