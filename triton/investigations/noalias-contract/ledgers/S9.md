## HOW-facts â€” each proven on real IR/asm

### FACT 1 â€” EMISSION: the A/not-A kill switch lands exactly one attr on arg %3 (block_tables)
Post-opt `.llir` `define`, arg %3 = `block_tables_ptr` (mapping: %0 output_ptr, %1 query, %2

`FunctionAttrs` infers `readonly captures(none)` on %3 **in BOTH** variants (the kernel never
writes through block_tables). The ONLY delta is the `noalias` token â€” the S4-canon kill switch.

### FACT 2 â€” the block-table load ALREADY selects a scalar in-loop `s_load_b32` at BASELINE
BASELINE amdgcn, inner tile loop `.LBB0_15` ("=>This Inner Loop Header: Depth=1"), first insn:
```

	.loc	1 2332 30                       ; mla.py:2332:30   <-- physical_block_idx = gl.load(...)

```
The line-mapped `s_load_b32` at `mla.py:2332:30` IS the block-table index load, selected as
**scalar SMEM inside the loop**, with the descriptor base in SGPRs (`s[38:39]`). No global/flat
load of the index; **in-loop `v_readfirstlane` = 0** already at baseline.

### FACT 3 â€” adding noalias produces ZERO instruction change (it is inert)
Full amdgcn diff (comments/labels stripped) BASELINE vs NOALIAS = **2 lines, both metadata**:
```
1280c1280
<       - .address_space:  global
---
>       - .actual_access:  read_only
```
This is the kernel-argument descriptor metadata for %3 flipping to reflect the read-only/noalias
annotation â€” NOT a single machine instruction. Load-kind histogram is **identical** in both:
`s_load_b128 x181, global_load_b128 x23, s_load_tr16 x16, s_load_b32 x4, s_load_b64 x1,
s_load_b512 x1, s_load_b256 x1`; in-loop `v_readfirstlane = 0` in both. The `global_load_b128`
x23 are the bf16 KV-cache data reads (legitimately vector); noalias does not (and should not)
touch those. **Delta: nothing scalarizes that was not already scalar; no churn is removed
because there was none.**

### FACT 3b â€” WHY noalias is inert here (the mechanism, S5 gate applied)
- Only writable ptr arg is `%0` (output_ptr, `writeonly`); EVERY read ptr incl. `%3` is
  `readonly`. There is **no in-loop store** that could clobber the block table.
- S5 gate (SIISelLowering.cpp:13148-13158): scalar SMEM fires on `uniform AND (invariant OR
  noclobber) AND align>=4 AND global/const AND scalarize-on`. Here: address uniform (`j` scalar
  loop var, block-uniform base); align 4 (i32, divisibility=16); **`readonly` alone makes the MMO
  invariant** â†’ the "invariant OR noclobber" leg is satisfied WITHOUT noalias. This is exactly S5
  Fact E corollary (`/tmp/s5/uniformity.ll @i32_noinv_noalias`: readonly-only still `s_load_dword`)
  and S4 Hop1 (readonly inferred in both states). `!amdgpu.noclobber` count in the module = 0 â€”
  the scalar selection goes through the invariant(readonly) path, not the noclobber(noalias) path.
- Contrast with F1 (why noalias WAS the switch there): MoE's gather load was (a) sub-dword i16 and
  (b) sat in a loop alongside an in-loop store (TDM-descriptor VGPR churn) whose may-alias AA could
  not disprove without noalias â†’ noclobber was the only route to invariance and noalias supplied
  it. mla has neither condition (i32, readonly, no aliasing in-loop store) â†’ nothing for noalias
  to disambiguate â†’ inert.

### FACT 4 â€” pa_decode_sparse does NOT compile at this SHA (pre-existing API drift, not noalias)

OLDER `async_gather(desc, indices, y_offset, dst)` API (cf. the nvidia/tutorial 4-arg form). This

## S3 soundness applied (shippability)

Per S3, `noalias_args=["block_tables_ptr"]` is a two-part caller promise: (1 readonly) the kernel
never writes through block_tables â€” TRUE (arg is `readonly`, only read); (2 noalias) block_tables
does not overlap any writable arg â€” the only writable arg is `output_ptr`, which is a distinct
output tensor from the block-table tensor in every caller (`mla_prefill_fwd`), so the promise
holds. So the annotation would be SOUND if applied. BUT: since Fact 3 shows it produces **zero asm
change** (the win is already captured by inferred `readonly`), there is **no benefit to ship it** â€”
it only adds contract surface (and S3's silent-miscompile UB exposure if a future caller ever
aliases output/block_tables) for no codegen gain. **Ship recommendation: DO NOT annotate the F2
block-table pointers** â€” the scalarization is already optimal at baseline; noalias buys nothing and
adds a soundness obligation.
