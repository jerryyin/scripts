## HOW-facts (real IR / asm — not paraphrased)

### 1. Every pointer is consumed only by a tensor descriptor; all traffic is TDM (TTGIR)
`BASE.ttgir` — each arg pointer flows into `tt.make_tensor_descriptor`, and all
data movement is TDM async DMA to/from LDS, NOT a raw-pointer load/store:
```
%x1_desc_0   = tt.make_tensor_descriptor %x1_ptr,   [%M,%N], [...]
%w1_desc     = tt.make_tensor_descriptor %w1_ptr,   [%N],    [...]
%res1_desc_1 = tt.make_tensor_descriptor %res1_ptr, [%M,%N], [...]
%out_res1_desc_2 = tt.make_tensor_descriptor %out_res1_ptr, ...
%out1_desc_3 = tt.make_tensor_descriptor %out1_ptr, ...
%0 = amdg.async_tdm_copy_global_to_local %0 into %smemX1   ...
%3 = amdg.async_tdm_copy_global_to_local %2 into %smemRes1 ...
%5 = amdg.async_tdm_copy_global_to_local %4 into %smemW1   ...
%9 = amdg.async_tdm_copy_local_to_global %8 from %smemOutRes1 ...
%13 = amdg.async_tdm_copy_local_to_global %12 from %smemOut1  ...
```
TTGIR op histogram: `3× async_tdm_copy_global_to_local`, `2× async_tdm_copy_local_to_global`,
`5× make_tensor_descriptor`, `3× local_load`, `2× local_store`, `3× tdm_wait`.
**Zero `tt.load` / `tt.store` / `global_load` of raw-pointer data.**

### 2. amdgcn: no data-dependent global/vector load; only kernarg s_loads + TDM + LDS
`BASE.amdgcn` memory-op histogram:
```
3 tensor_load_to_lds       ; TDM: x1, res1, w1  -> LDS
2 tensor_store_from_lds    ; TDM: out_res1, out1 <- LDS
3 ds_load_b128 / 2 ds_store_b128   ; LDS traffic (already 128-bit wide)
1 s_load_b256 / 1 s_load_b128 / 1 s_load_b96 / 1 s_load_b64   ; KERNARG only
```
All 4 `s_load`s read from `s[0:1]` (kernarg segment ptr, addrspace 4) at fixed
offsets `0x0,0x20,0x28,0x38` — they load the kernel *arguments* (the pointer values,
strides, eps) from the read-only kernarg segment, which is uniform by construction
and independent of any `noalias` on the pointer values. `.kernarg_segment_size: 88`.
**No `global_load`, no `v_readfirstlane` anywhere** — the F1 gather symptom cannot
appear because no raw-pointer scalar load exists.

### 3. The kill-switch: noalias applied, output identical, asm bitwise identical
`NA.llir` post-opt define proves the annotation took effect:
```
@_gluon_fused_rms_kernel(ptr addrspace(1) noalias %0, ... noalias %1, noalias %2,
                         noalias %3, noalias %4, float %5, ...)
```
- FFM correctness (both variants, M=2 N=32, has_res): `OUT max abs diff: 0.0`,

- **amdgcn diff BASE vs NA: identical.** `md5sum` both = `8e683109c26c2192430ba5e0e449514c`;

- **LLVM IR body diff (ignoring the define line): EMPTY** — the only difference in

- `grep -c` for `invariant.load|amdgpu.noclobber|alias.scope|noalias`(metadata) in

## Hypotheses + refutation trail

- **H1 (from F1/S4): noalias→readonly→noclobber→scalar s_load hoist.**
  REFUTED for F5. That chain requires a uniform+invariant raw-pointer `global_load`
  in a loop. F5 has no raw-pointer load at all (TDM DMA only) and no loop over the
  pointer data; `AMDGPUAnnotateUniformValues.visitLoadInst` has nothing to tag.
  Confirmed: no `!amdgpu.noclobber`, no `s_load` of *data* (only kernarg s_loads).
- **H2 (from S7, the required lens): noalias unblocks LoadStoreVectorizer /
  SILoadStoreOptimizer widening.** REFUTED for F5. LSV/SILoadStoreOptimizer widen
  adjacent scalar/vector *memory ops* whose reordering is blocked by a may-alias
  intervening access. F5's bulk loads/stores are single TDM descriptor copies
  (already whole-tile), and the LDS ops are already `ds_load_b128`/`ds_store_b128`
  (max width). There is no pair of narrow raw-pointer accesses separated by an
  aliasing store to merge. Width histogram is identical BASE vs NA.
- **H3: some scalar-param pointer could benefit.** REFUTED — there are none;
  eps1/strides are by-value scalars (i32/float in the signature).
- **H4: maybe FunctionAttrs would infer readonly and trigger a later opt (as in F1).**
  REFUTED — post-opt IR bodies are identical; even the inferred-attr surface didn't
  change any instruction. (Note: unlike F1 where noalias re-surfaced as inferred
  `readonly`, here the post-opt define still shows bare `noalias`; either way, no
  downstream op reacted.)

## CLAIM STATUS

**Answer:** noalias on the norm kernel's aux/data pointers produces **no asm delta**
— the amdgcn is bitwise identical (same md5), and the only change in the whole
post-opt LLVM module is the `noalias` token on the arg list. This is a **grounded
negative**, not a forced result: the mechanism is that F5 routes every pointer
through `tt.make_tensor_descriptor` and moves all data via TDM
(`tensor_load_to_lds`/`tensor_store_from_lds`) into LDS, so neither of the two
win-mechanisms proven upstream (F1/S4 scalarization, S7 vectorizer/coalescer
widening) has an eligible raw-pointer memory op to act on. FFM-correct at M=2,N=32
in both states (rel diff 0.0).

**Counter-experiment that would refute "no delta":** exhibit, within F5, a
raw-pointer `tt.load`/`tt.store` (or a data-dependent `global_load` in a loop) whose
lowering flips under noalias. Result: none exists — TTGIR shows 0 `tt.load`/`tt.store`
and the amdgcn shows 0 `global_load`/`global_store`; the toggle changed 0 instructions.
The complementary positive control already exists (F1/S4: the same toggle on a
raw-pointer gather load flips 16 `v_readfirstlane`→0 and selects `s_load_u16`),
confirming the toggle is live and the null result is specific to F5's TDM data path.

## Soundness note (S3 applied)

Per S3, the caller contract is only sound if the noalias'd pointer is never written
through in-kernel AND never aliases a writable arg. In F5 the aux *reads* (w1_ptr,
res1_ptr, x1_ptr) are distinct torch tensors from the outputs (out1, out_res1 are
`torch.empty`), so noalias on the *read* pointers would be honestly assertable.
noalias on the *output* pointers (out1_ptr, out_res1_ptr) is also honest here since
each is a separate allocation. But since the asm delta is zero, there is **no reason
to ship any noalias annotation on F5** — it buys nothing and only adds a caller
obligation (S3 silent-miscompile surface) with no upside. **Recommendation: no-ship**
for F5.
