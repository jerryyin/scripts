## HOW-facts (real IR/asm + exact code path)

### Fact 1 — noalias alone does NOT change global_load ⇒ buffer_load at ISel
Isolated `llc` on two identical addrspace(1) kernels differing only by `noalias`
(`/tmp/td-S6/probe_load.ll`, `-mcpu=gfx1250 -O3`):
```
=== load_plain ===    global_load_b32 v1, v0, s[0:1] scale_offset ; global_store_b32 ...
=== load_noalias ===  global_load_b32 v1, v0, s[0:1] scale_offset ; global_store_b32 ...
```
Identical. The addressing mode (flat/global vs MUBUF/buffer) is fixed by pointer
*address space* + addressing, not by any pointer alias attribute.

### Fact 2 — what ACTUALLY selects buffer_load / s_buffer_load (the required inputs)
Isolated `llc` (`/tmp/td-S6/probe_buffer.ll`):
- `s_buffer_load_b32 s2, s[8:11], 0x0` ← from the `llvm.amdgcn.s.buffer.load.f32`
  **intrinsic** + a `<4 x i32>` resource descriptor.
- `buffer_load_b32 v1, v1, s[8:11], null offen` ← from a plain load through an
  **addrspace(7)** fat buffer pointer (yields the `s[8:11]` descriptor).
Neither path involves `noalias`. Buffer selection requires either the intrinsic or an
addrspace(7)/addrspace(8) buffer-resource pointer — an explicit representation change,
not an attribute.

### Fact 3 — the ONLY LLVM AMDGPU codegen pass touching buffers is fat-ptr lowering
Full gfx1250 -O3 codegen pass list (`llc -debug-pass=Structure`,
`/tmp/td-S6` run): the single buffer-related pass is
`amdgpu-lower-buffer-fat-pointers` ("Lower buffer fat pointer operations to buffer
resources") — it lowers **addrspace(7) → addrspace(8)** buffer ops. There is **no**
pass named/behaving like "promote-global-to-buffer" or "infer-buffer-from-noalias".
The AMDGPU alias-analysis passes present (`amdgpu-aa`, `basic-aa`,
`scoped-noalias-aa`) feed GVN/LICM/scheduling — NOT address-space or addressing-mode
selection. So a dedicated LLVM pass keyed on aliasing does **not** exist and is not on
the path.

### Fact 4 — buffer_load on gfx1250 is a TRITON MLIR-level decision, and it is
### gated on 32-bit-offset provability, with ZERO alias input
`third_party/amd/backend/compiler.py:303` runs, when `knobs.amd.use_buffer_ops`:
`add_canonicalize_pointers` → `add_convert_to_buffer_ops` → `add_optimize_buffer_op_ptr`.
The decision lives in `ConvertToBufferOps.cpp::canUseBufferOps` (lines 135-173):
1. base is uniform (splat) + non-uniform offset (`tt.addptr` of a `tt.splat`),
2. offset bit width ∈ {32, 64},
3. **`isSafe`** = `isFuncArgWith32bitPtrRange(base)` (the `tt.pointer_range=32` arg
   attr) **OR** `isByteOffsetSmallerThan2GB(...)` (range analysis).
`grep -c noalias|mayAlias|aliasInfo ConvertToBufferOps.cpp` → **0**. Aliasing is never
consulted. The `tt.pointer_range=32` attr is derived from
`HIPBackend.is_within_2gb(arg)` → `"S"` specialization → `parse_attr` — an
allocation-size guarantee, orthogonal to `noalias` (which S1 showed comes from a
disjoint opt-in path, `noalias_args`→`tt.noalias`→`llvm.noalias`).

### Fact 5 — A/not-A kill-switch (decisive), one module, dynamic-i64 offset so only
### the guarantee decides (`/tmp/td-S6/probe2.ttgir`, `--tritonamdgpu-convert-buffer-ops=gfx-arch=gfx1250`)
```
@plain_dyn    (no attrs)              -> tt.load            (NOT converted)
@noalias_dyn  (tt.noalias=1 only)     -> tt.load            (NOT converted)   <-- noalias does nothing
@range_dyn    (tt.pointer_range=32)   -> amdg.buffer_load   (converted)       <-- range is what fires it
```
This is the essentiality test: adding noalias flips nothing; adding pointer_range flips
global→buffer. (In an earlier probe with a *statically small* range 0..256, even the
noalias-only and plain cases converted — because `isByteOffsetSmallerThan2GB` proved
the offset independently — which only reinforces that provable-range, not noalias, is
the driver.)

### Fact 6 — Triton emits VECTOR buffer_load, never s_buffer_load
`amdg.buffer_load` lowers via `BufferOpsEmitter.cpp:120` to `ROCDL::RawPtrBufferLoadOp`
= `llvm.amdgcn.raw.ptr.buffer.load`. Isolated `llc` on that intrinsic
(`/tmp/td-S6/rawbuf.ll`) → `buffer_load_b32 v0, v0, s[0:3], null offen`. There is **no
`s_buffer_load`/`s.buffer.load` emission anywhere in `third_party/amd/lib`** (grep = 0
hits outside `supportsBufferLoadToLocal` naming). s_buffer_load only arises from the
scalar intrinsic (not emitted by Triton) or backend scalarization of a buffer-resource
load. Note: the F1 prefill win is `s_load` (scalar *global*), a distinct instruction —
not `s_buffer_load` — so the two mechanisms must not be conflated.

## Hypotheses + refutation trail

- **H1 (Yin's hypothesis): "noalias lets AMDGPU pick buffer/s_buffer loads."**
  REFUTED — Fact 1 (isolated llc: noalias changes nothing), Fact 4 (Triton pass has no
  alias input), Fact 5 (noalias-only case stays tt.load). noalias is inert for buffer
  selection.
- **H2: "A dedicated LLVM pass is required (and could be keyed on noalias)."**
  PARTIALLY: a dedicated *representation* is required, but not an alias-keyed pass and
  not at LLVM level for Triton. Fact 3: the only LLVM buffer pass lowers pre-existing
  addrspace(7) fat pointers; it does not synthesize buffers from addrspace(1)+noalias.
  Fact 2: buffer_load needs an addrspace(7)/(8) pointer or the s.buffer intrinsic —
  an explicit IR change. On gfx1250 that explicit change is made **in Triton MLIR**
  (`convert-to-buffer-ops`), keyed on offset range, before LLVM ever runs.
- **H3: "pointer_range=32 (2GB) is the real gate."** SUPPORTED — Fact 4 (code),
  Fact 5 (range-only case converts, noalias-only does not).

## CLAIM STATUS

**CLAIM (answered, irrefutable at ISel + MLIR + plumbing levels):**
- **Does noalias enable global_load⇒buffer_load/s_buffer_load?** No.
  - **noalias wrt buffer_load: NEITHER necessary NOR sufficient.** Not sufficient
    (Fact 1, Fact 5 @noalias_dyn stays tt.load). Not necessary (Fact 5 @range_dyn
    converts with no noalias; the whole AITer gemm buffer story runs on pointer_range).
- **Is a dedicated pass/intrinsic required?** Yes — but an *offset-range-gated
  representation change*, not an alias-driven one:
  - buffer_load requires an addrspace(7)/(8) buffer pointer or the
    `raw.ptr.buffer.load` intrinsic (Fact 2, 6);
  - on gfx1250 Triton performs that change at **MLIR** via
    `tritonamdgpu-convert-buffer-ops`, gated on `tt.pointer_range=32` / 2GB range
    analysis (Fact 4);
  - the only LLVM-level buffer pass (`amdgpu-lower-buffer-fat-pointers`) merely lowers
    an *already-present* fat pointer (Fact 3).
- **s_buffer_load specifically:** not emitted by Triton at all (Fact 6); only via the
  scalar intrinsic path. Out of reach of any noalias contract on the current backend.

**Counter-experiment that would refute (and its result):**
- *"Give a load noalias-only (no pointer_range, unprovable offset) and see buffer_load
  appear."* → Fact 5 @noalias_dyn: stays `tt.load`. Not refuted.
- *"Find an alias/noalias check inside the buffer-conversion decision."* →
  `grep -c` in ConvertToBufferOps.cpp = 0; canUseBufferOps consults only
  splat/offset-width/range. Not refuted.
- *"Find an LLVM AMDGPU pass that promotes addrspace(1)+noalias to a buffer resource."*
  → codegen pass list has only `amdgpu-lower-buffer-fat-pointers` (lowers existing
  addrspace(7)); none synthesize from addrspace(1). Not refuted.
