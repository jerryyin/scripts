## HOW-facts (real IR/asm lines)

### Fact 1 — the contract plumbed correctly (ON has noalias, OFF does not)
TTIR (ON): `%cache_indices_ptr ... {tt.noalias = 1}`, `%has_initial_states_ptr ... {tt.noalias = 1}`,
`%query_start_loc_ptr ... {tt.noalias = 1}`. TTIR (OFF): grep `tt.noalias` → empty.
Post-opt LLVM `define` line — the ONLY difference is `noalias` on args %4/%5/%6:
```
ON : ... ptr addrspace(1) noalias nofree readonly captures(none) %4,   <- cache_indices
         ptr addrspace(1) noalias nofree readonly captures(none) %5,   <- has_initial_states
         ptr addrspace(1) noalias nofree readonly captures(none) %6, ... <- query_start_loc
OFF: ... ptr addrspace(1)         nofree readonly captures(none) %4,
         ptr addrspace(1)         nofree readonly captures(none) %5,
         ptr addrspace(1)         nofree readonly captures(none) %6, ...
```
**Both are `readonly`** (LLVM-inferred regardless of noalias) — matches S3 Fact 1 (readonly ⟂ noalias).

### Fact 2 — the asm instruction stream is BYTE-IDENTICAL ON vs OFF (the headline)
`.text` md5 (instructions + directives, kernarg-metadata YAML stripped): **`779cabac97a506207b0d3c4402f35969` for BOTH**. `diff` of the text sections = empty ("IDENTICAL TEXT SECTION").
Full-file md5 differs (`c425b8...` vs `a4465e0...`) and line count 913 vs 910 ONLY because the ON
kernarg-metadata YAML adds 3 lines:
```
> - .actual_access:  read_only    (x3, on the cache_indices / has_initial_states / query_start_loc args)
```
These are HSA kernarg-descriptor annotations — informational, zero executed instructions.

### Fact 4 — the two int32 index loads are ALREADY scalar s_load at baseline (OFF)
gfx1250 asm prologue (OFF, .amdgcn):
```
.loc 1 70   s_load_b64 s[20:21], s[4:5], 0x0    ; query_start_loc_ptr[idx_seq] & [idx_seq+1] coalesced
.loc 1 88   s_load_b32 s1,       s[0:1], 0x0    ; cache_indices_ptr[idx_seq]
```
Both are scalar SMEM `s_load` with NO contract. In post-opt `.llir` these are plain
`%30 = load <1 x i32>, ptr addrspace(1) %29 ... !dbg (line 70)`, `%47 = load <1 x i32> ... (line 88)`
— **byte-identical in ON and OFF**, and `grep amdgpu.noclobber .llir` = **0 in BOTH**. So `noalias`
produced NO `!amdgpu.noclobber` here. Why they are s_load anyway: the address derives from
`tl.program_id(0)` (uniform; lowered from `ttmp9`/`ttmp7` scalar hw-regs), and the loads sit in the
prologue BEFORE any store, so `readonly` alone makes the MMO invariant — satisfying the
"invariant OR noclobber" leg of the S5 gate (SIISelLowering.cpp:13148-13158) WITHOUT needing the
noalias-derived noclobber. This is S5 Fact E's exact corollary ("readonly alone makes the MMO
invariant… the win tracks noalias only where a store can otherwise clobber"). Here nothing clobbers,
so noalias is redundant.

### Fact 5 — the i8 has_initial_states load: sub-dword ceiling, not aliasing (S5 Fact B/C)
`.loc 1 111  global_load_u8 v5, v10, s[2:3]` then `v_readfirstlane_b32 s2, v5`. In `.llir`:
`%59 = load <1 x i8>, ptr addrspace(1) %58, align 1` — identical ON/OFF. It stays VMEM because an
i8 has no wide-SMEM form (S5 Fact B: sub-dword vector SMEM does not exist) and here it is not even
aligned to the scalar-extload path; noalias is irrelevant to the ceiling. The single
`v_readfirstlane_b32` (asm line 145) is in block `.LBB0_5` (chunk_offset==0 init-state prep),
NOT in the compute inner loop `.LBB0_16` ("=>This Inner Loop Header"). So there is ZERO in-loop
index-load activity at baseline — no F1-style `v_readfirstlane`-in-loop to kill.

### Fact 6 — S3 soundness applied (contract is sound but inert)
If the annotation WERE ever needed it would be sound: `query_start_loc`/`cache_indices`/
`has_initial_states` are genuine read-only metadata arrays, never stored through in-kernel (the
kernel's only stores are to `conv_states` (%3) and `o_ptr` (%7), distinct writable args), and they
do not overlap those writable args — both S3 axes hold. But since the codegen is byte-identical
(Fact 2), the contract carries no benefit to weigh against the S3 silent-miscompile risk: it is
pure caller-side risk for zero codegen gain. NO-SHIP.

### Fact 7 — FFM correctness
The `.text` instruction stream is byte-identical ON==OFF (Fact 2, same md5), so ON and OFF are the
SAME machine code and produce identical results BY CONSTRUCTION — the S13-style asm-identity proof,
which is strictly stronger and noise-free vs a numeric FFM compare. A live FFM run was NOT performed:
