## HOW-facts — proven on the REAL decode kernel + the standalone reproducer

### FACT 1 — the residual is NOT load-selection; the noalias contract IS working
Index-load histogram in the shipped (contract-ON) decode kernel (`decode_stock56.s`):
```
s_load_u16      : 16   (scalar SMEM index loads — the #1885/S4 prefill mechanism, firing in decode)
global_load_u16 :  0
```
So S4/S5's `noalias => s_load_u16` chain DOES fire in decode. The 8 residual `v_readfirstlane`
are therefore a DIFFERENT thing from the load — they are not touched by scalarizing the load.

### FACT 2 — the 8 residual rfl lift a loop-invariant descriptor into scalar operands
Stock decode K-loop body (`.LBB0_5`, lines 1006..1219 of `decode_stock56.s`):
```asm
.LBB0_5:                                ; =>This Inner Loop Header: Depth=1
	v_readfirstlane_b32 s16, v34
	v_readfirstlane_b32 s17, v1
	v_readfirstlane_b32 s18, v36
	v_readfirstlane_b32 s19, v35
	v_readfirstlane_b32 s56, v38
	v_readfirstlane_b32 s57, v37
	v_readfirstlane_b32 s58, v40
	v_readfirstlane_b32 s59, v39
	tensor_load_to_lds s[8:11], s[36:43]
	tensor_load_to_lds s[60:63], s[44:51], s[16:19], s[56:59]   ; row-index descriptor operand (SCALAR)
	tensor_load_to_lds s[8:11], s[24:31]
	s_cbranch_scc1 .LBB0_5
```
The 8 rfl produce exactly the two `<4 x i32>` scalar descriptor groups `s[16:19]` and `s[56:59]`
that feed the gather `tensor_load_to_lds`. Their source VGPRs are defined in the PREHEADER:
`v36,v35,v38,v37,v40,v39` at lines 757..762, `v1` at line 64 (`v34` similarly pre-loop) — all
before the K-loop header (1006). => the broadcasts are loop-invariant.

### FACT 3 — the loop is UNIFORM (EXEC never written), so the hoist is legal
In the stock K-loop (1006..1219) NO instruction writes EXEC (`exec` as destination): grep for a
dest-position `exec` => 0 hits. The sole `exec` mention is a READ:
`s_and_b32 s26, s92, exec_lo` (exec_lo is a source; s26 is the dest). EXEC is loop-invariant =>
"first active lane" is identical every iteration => the convergent `readfirstlane` result does not
change => safe to hoist. (This is exactly the `@bug` shape of the standalone reproducer.)

### FACT 4 — stock leaves it at 8 on BOTH pins (longstanding, not a version effect)
Same real decode `.llir`, three llc:
```
stock 56421f92 : in_loop_rfl = 8
stock 62b7cf96 : in_loop_rfl = 8
patched        : in_loop_rfl = 0
```
Two stock pins agree at 8 => holding the IR constant, the LLVM version changes nothing; this is a
longstanding missed optimization, not a codegen regression. (Matches the reproducer's
"same IR => same count on both pins".)

### FACT 5 — the patch HOISTS (does not delete); asm is numerically inert
Patched decode K-loop body (`decode_patched.s`, 1015..1218):
```asm
	tensor_load_to_lds s[8:11], s[36:43]
	tensor_load_to_lds s[60:63], s[44:51], s[16:19], s[56:59]
	tensor_load_to_lds s[8:11], s[24:31]
	s_cbranch_scc1 .LBB0_5
```
The 8 rfl are gone from the loop; they now sit in the preheader at lines 976..986:
```asm
	v_readfirstlane_b32 s16, v6
	v_readfirstlane_b32 s17, v5
	v_readfirstlane_b32 s18, v8
	v_readfirstlane_b32 s19, v7
	v_readfirstlane_b32 s56, v4
	v_readfirstlane_b32 s57, v1
	v_readfirstlane_b32 s58, v2
	v_readfirstlane_b32 s59, v3
```
Numeric-equivalence proof at asm level:
- **Total per-kernel rfl preserved: stock 19, patched 19** => the 8 were MOVED, not removed.
- **Descriptor-feeding op byte-identical** in both:
  `tensor_load_to_lds s[60:63], s[44:51], s[16:19], s[56:59]` (and the second group
  `s[52:59], s[36:39], s[44:47]`) appear identically in stock and patched.
This is pure loop-invariant code motion of a value proven loop-invariant in a uniform loop — the
result the loop consumes is unchanged. (FFM on gfx1250 exercises the triton-built compiler, not
standalone llc, so the byte-identical asm + identical consumed operands is the correctness
evidence for the llc-level change; the contract-ON kernel itself is FFM PASS, FACT-sanity.)

### FACT 6 — targeted & correct: kill-switch + divergent-safety double dissociation
- **Patched llc, `-amdgpu-hoist-uniform-readfirstlane=false`** on the same decode `.llir`:
  `in_loop_rfl = 8` (reverts to stock). => the flag/hoist is THE mechanism, not some other
  build delta.
- **Standalone reproducer** (`amdgpu_readfirstlane_licm/ir/repro.ll`), patched llc, flag on:
  `@bug (uniform) = 0`, `@safe (divergent) = 4` (unchanged). The divergent loop (EXEC redefined
  inside) correctly KEEPS its broadcasts. `reproduce.sh fixed` reports "FIX confirmed:
  @bug hoisted to 0; @safe (divergent) unchanged => correct and targeted."

## Hypotheses + refutation trail

- **H1 "decode's 8 rfl are the same load-selection problem as prefill, so the noalias contract
  should also fix them."** REFUTED — decode's index load is ALREADY `s_load_u16` (x16, zero
  global_load) with the contract on (FACT 1), yet 8 rfl remain. They lift a preheader-built

  (FACT 4).
- **H3 "the patch deletes the readfirstlane / changes results."** REFUTED — total rfl 19->19

- **H4 "the hoist is unsafe (readfirstlane is convergent)."** REFUTED for THIS loop — EXEC is
  never written in the loop (uniform, FACT 3); the patch gates hoisting on EXEC loop-invariance
  via `shouldAnalyzePhysregInMachineLoopInfo`, and the divergent control `@safe` is correctly
  left at 4 (FACT 6).
- **H5 "some other build difference in the patched llc explains the 0."** REFUTED — the same
  patched binary with the guard flag OFF reproduces 8 (FACT 6 kill-switch).

## CLAIM STATUS

**CLAIM (answered):**
1. The decode residual is **loop-invariant-convergent descriptor lift**, not load-selection: the
   noalias contract already scalarizes the index load (`s_load_u16` x16), and the 8 rfl broadcast
   a preheader-built `<4 x i32>` descriptor into the scalar operands of `tensor_load_to_lds` in a
   **uniform** loop (FACTs 1-3).
2. Under the shipped noalias contract the residual is **8 on both stock llc pins** (FACT 4).
3. The MachineLICM patch (`db4972674`, default-on flag) takes the real decode kernel to **0** by
   **hoisting** the 8 into the preheader with total-rfl preserved and byte-identical descriptor
   operands (FACT 5), and is **correct** (divergent `@safe` kept at 4) and **targeted**
   (flag kill-switch reverts to 8) (FACT 6).

**SHIP DECISION: SHIP the LLVM patch (recommended), with 8 acceptable as a fallback.**
- *For shipping:* correct (uniform-only, EXEC-gated), numerically inert (pure hoist), targeted
  (a hook defaulting false + a guarded AMDGPU opt-in), and it closes decode to 0 to match prefill
  — completing the #1885 story (prefill=triton contract, decode=this LLVM patch; orthogonal).
- *Magnitude / why 8 is acceptable if upstreaming stalls:* the win is MODEST. The 8 hoisted ops
  are single-cycle SALU `v_readfirstlane`, ~5.7% of a ~141-instruction decode K-loop body that is
  dominated by 3 async `tensor_load_to_lds` (TDM) + MFMA compute; it also trims in-loop SGPR
  liveness. Real gfx1250 timing is not measurable under FFM, and this HOST is gfx950 (a separate
  arch); no runtime number is claimed. So the recommendation rests on correctness + asm delta,
  not a measured speedup — do NOT gate the decode ship on the patch; it is a clean, upstreamable
  incremental improvement, not a blocker.

**Counter-experiments that would refute (and their results):**
1. "Show the 8 rfl are the index load, so the contract should remove them." -> load is s_load_u16
   x16, global_load_u16=0; rfl still 8. Not refuted.
2. "Show the patch changes numerics (deletes rfl / different operands)." -> total rfl 19->19,
   descriptor ops byte-identical. Not refuted.
3. "Show the hoist is unsafe on a divergent loop." -> @safe stays 4 under patched llc. Not refuted.
4. "Show the 0 is from an unrelated build diff, not the hoist." -> patched llc + flag=false => 8.
   Not refuted.
5. "Show it's an LLVM-version artifact." -> both stock pins = 8. Not refuted.
