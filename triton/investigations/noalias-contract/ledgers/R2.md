## HOW-facts (real evidence)

### Fact 1 — the harness is complete and correct up to GPU execution
Two files written and importable:
- `harness/gfx950_bench.py` — `bench_median(launch, n_reps=7, warmup=20)` →
  `BenchResult(median_ms,min_ms,max_ms,mean_ms,stdev_ms,spread_pct,samples_ms)`.
  Fail-fast: rejects `n_reps<5`, non-positive event times, and (via
  `_assert_device`) any device whose name is not MI35*/gfx950.
- `harness/bench_gmm.py` — per-kernel driver; `--shape M,K,N,G`, `--all`
  (sweeps `REAL_SHAPES` from test_gmm), default `SMALL_SHAPE=(4096,1408,2048,8)`,
  `--reps/--warmup/--work-stealing`.

The device-detection leg WORKS: the driver prints
`device='AMD Instinct MI355X'  HIP_VISIBLE_DEVICES=0  torch=2.11.0+rocm7.14...`
i.e. gpu-lock held, device pinned, MI355X correctly identified. The harness runs
right up to the first `torch.cuda` tensor allocation, then the torch image fails.

### Fact 2 — the grounded blocker: torch built for gfx1250 ONLY, not gfx950
```
torch build arch list: ['gfx1250']
device0: AMD Instinct MI355X          <-- gfx950
ver: 2.11.0+rocm7.14.0a20260623
```
`torch.cuda.get_arch_list()` returns **`['gfx1250']`** — the wheel contains ONLY
gfx1250 device code objects. The device is gfx950. So the FIRST kernel launch
(even a bare `torch.randn((1024,1024), device='cuda')`, no aiter, no triton)
fails:
```
torch.AcceleratorError: CUDA error: device kernel image is invalid   (hipErrorInvalidImage)
```
This reproduces identically (a) inside the gmm driver at `gen_gmm_tensors` →
`torch.randn`, and (b) with bare torch and no aiter import — so it is NOT an
aiter/triton library-order issue; it is a torch-vs-device arch mismatch.

### Fact 3 — `HSA_OVERRIDE_GFX_VERSION` does not bridge it (refutation of the obvious workaround)
`HIP_VISIBLE_DEVICES=0 HSA_OVERRIDE_GFX_VERSION=9.5.0 gpu-lock python3 -c 'torch.randn(...,device=cuda); x@x'`
→ same `hipErrorInvalidImage`. Expected: the override remaps *within* an existing
image family; there is no gfx950 code object in the wheel to remap to, so nothing
to select. Not a config fix.

### Fact 4 — no alternative gfx950 torch exists on the host
`pip list` → one torch. `find / -path '*torch*version.py'` → only the installed
package + `/root/triton/mi400/notorch/` (a stub, no CUDA). No conda envs, no
venvs, no cached wheels. There is no second interpreter/env with a gfx950 build.

## Hypotheses + refutation trail

- **H1 "aiter import corrupts the torch/rocm libs (LD_LIBRARY_PATH), fix import order."**
  REFUTED — Fact 2(b): bare torch with zero aiter import fails identically on the
  first launch.
- **H2 "wrong device pinned / permissions."** REFUTED — Fact 1: device correctly
  reports MI355X under the pin; allocation reaches the GPU; it is the *code object*
  that is invalid, not the device handle. (`/dev/kfd` access is fine — rocminfo +
  device_name work.)
- **H3 "HSA_OVERRIDE bridges gfx950↔gfx1250."** REFUTED — Fact 3.
- **H4 "torch just needs recompiled kernels at runtime (like triton)."** REFUTED —
  torch ships prebuilt device code (get_arch_list is the manifest); it does not
  JIT for an absent arch. Triton *would* JIT for gfx950, but every triton kernel
  is launched *through* torch tensors, which die first.
