# The two root causes (traced, not paraphrased)

## RC1 — async_gather API drift (triton PR #61, "make async_tdm_gather/scatter pure")
Original signature (git-confirmed, PR #9313 introduced it):
`async_gather(desc, src_row_indices, src_col_offset, dst, mbarrier=None)` — 5 positional-eligible.
PR #61 REMOVED `src_col_offset` (the column offset is now carried by the descriptor via
`update_tensor_descriptor(desc, add_offsets=[0, col])`). Current signature (tdm.py:421):
`async_gather(desc, src_row_indices, dst, mbarrier=None)`.

Both kernels were written against the OLD signature, passing `src_col_offset=0`:
- pa_decode_sparse.py:301 `async_gather(kv_desc, safe_slot_cur, 0, kv_bufs.index(0))`
- pa_decode_sparse.py:341 `async_gather(kv_desc, safe_next_slot, 0, kv_bufs.index(async_idx))`
- fused_kv_cache.py:107 `_issue_tdm_gather_2d`: `async_gather(desc, off_0, 0, smem)`

Under the new 4-arg API the literal `0` binds to `dst` and the smem buffer binds to `mbarrier`;
the builtin then executes `dst_ndim = len(dst.shape)` on the constexpr `0` -> the exact failure.

**Faithfulness argument (no semantic guesswork):** every call passed `src_col_offset=0` (zero
column offset), and each descriptor is built at column 0 — `kv_desc` in pa_decode
(`make_tensor_descriptor(base=unified_kv_ptr, shape=[total_pages, BLOCK_D], ... block_shape=[BLOCK_K, BLOCK_D])`,
no non-zero column position), and the reshape cos/sin descriptors likewise start at column 0.
So dropping the `0` argument is the *identity* migration the PR prescribes — the gather still
reads from column 0. No `update_tensor_descriptor` is needed because there was no non-zero offset
to carry.

## RC2 — MLA-cat launcher off-by-one drops kv_cache_stride_d
Kernel `_fused_qk_rope_cat_and_cache_mla_kernel` has 47 non-constexpr positional params; the arg
after `num_decode_toks_for_zeros` is `MAX_EMBD_POS` ("unused here; kept for a uniform launch with
the BLOCK kernel"). The MLA-cat launcher (`fused_qk_rope_cat_and_cache_mla`, line ~274) jumped
straight from `num_decode_toks_for_zeros` to `*q_nope.stride()`, supplying NO value for
`MAX_EMBD_POS`. Every subsequent stride then shifts one slot left, and the 47th param
`kv_cache_stride_d` falls off the end -> `TypeError: dynamic_func() missing 1 required positional
argument: 'kv_cache_stride_d'`.

**Faithfulness argument:** the sibling reshape launcher (`fused_qk_rope_reshape_and_cache`, line
~557) already passes `max_embd_pos = cos.shape[0]` for the same `MAX_EMBD_POS` slot. The fix mirrors
that exactly: insert `cos.shape[0]` between `num_decode_toks_for_zeros` and `*q_nope.stride()`.
`MAX_EMBD_POS` is unused inside the MLA kernel body, so the value is inert to kernel semantics; using
`cos.shape[0]` matches the established convention rather than inventing a placeholder. This keeps the
"uniform launch" contract the kernel comment documents (no kernel-signature change needed).
