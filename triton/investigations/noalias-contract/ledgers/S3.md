## The exact contract statement (deliverable)

A kernel author annotating pointer arg `P` with `noalias_args=["P"]` (and, by convention,
typing it `tl.const`) MUST guarantee, for the entire kernel dispatch:

1. **readonly axis** ŌĆö the kernel never writes through `P` or any pointer derived from `P`
   (`tl.const` enforces this in the Triton frontend: `store` through a const pointer is a
   compile error). This is what makes the load *invariant/reusable* and is what LLVM keys
   `s_load` selection off of.
2. **noalias axis** ŌĆö no location reachable through `P` is *modified* during the dispatch
   via any pointer **not based on `P`** ŌĆö i.e. `P`'s buffer does not overlap any *writable*
   kernel argument (Y/out/scratch/ŌĆ”). Overlap with other *read-only* args is harmless.
3. **hardware-coherence corollary (gfx1250-specific)** ŌĆö because the value will be fetched
   through the **scalar cache (K$)** via `s_load`, and the scalar path is not kept coherent
   with in-kernel vector-memory writes without an explicit invalidate, ANY in-kernel vector
   store into `P`'s buffer (even a "legal" self-write) would be invisible to the scalar read.
   The readonly promise (1) already forbids this; the corollary is why violating it is not
   merely a stale-CSE hazard but also a cache-coherence hazard.

Violation of (1) or (2) is **undefined behavior** (LangRef: the noalias guarantee "only
holds for memory locations that are modified ŌĆ” during the execution of the function. If
there are other accesses not based on the argument ŌĆ” the behavior is undefined."). The
manifestation here is a **silent miscompile**, not a trap.

## HOW-facts (evidence)

### Fact 1 ŌĆö the two axes are ORTHOGONAL, proven by FunctionAttrs (opt)
`opt -passes=function-attrs` on the minimal gather model (`/tmp/td-S3/gather_{noalias,plain}.ll`,
kernel loads through `%idxp`, stores through `%out`):

```
NOALIAS: define ...@gather(ptr addrspace(1) noalias nofree readonly captures(none) %idxp, ptr addrspace(1) nofree writeonly captures(none) %out, i32 %n)
PLAIN  : define ...@gather(ptr addrspace(1)         nofree readonly captures(none) %idxp, ptr addrspace(1) nofree writeonly captures(none) %out, i32 %n)
```
- `readonly` is inferred **in BOTH** variants ŌĆö it depends ONLY on the fact that the function
  never stores through `%idxp`, NOT on `noalias`. This is the mechanical proof that
  **readonly Ō¤é noalias**.
- Confirms S1 Fact 1/Fact 5: the AMD backend (`handlePointerContractArgs`,
  `Utility.cpp:2244`) emits ONLY `llvm.noalias` from `tt.noalias`; it emits NO `llvm.readonly`.
  And `tl.const` (core.py:146 `class const`, pointer `.const` flag) is a frontend-only
  store-prohibition ŌĆö it does not become an LLVM attr. So in the real kernel, `readonly`
  can *only* come from FunctionAttrs inference, exactly as reproduced here.

### Fact 2 ŌĆö noalias is the switch that lets LICM hoist the index load (opt -O3)
Same two models after `-O3`:
- **noalias:** the index load is hoisted to `entry`, carrying AMDGPU metadata:
  `%idx = load i32, ptr addrspace(1) %idxp, align 4, !amdgpu.noclobber !0` ŌĆö computed ONCE.
- **plain:** the load stays INSIDE `%loop` and is even unrolled ŌĆö reloaded every iteration
  (`%idx`, `%idx.1`, `%idx.2`, ŌĆ”). Reason: the store `%out[idx]` **may-alias** `%idxp`, so
  LICM/GVN must assume the store clobbers it. This is the may-alias pessimism S1/T0 describes.
- `!amdgpu.noclobber` is set by AMDGPUAnnotateUniformValues when AA proves the location is
  not modified in the function; noalias is precisely the AA fact that makes that provable.

### Fact 3 ŌĆö the hardware effect: scalar s_load vs in-loop vector global_load (llc gfx1250)
`opt -O3 | llc -mcpu=gfx1250` on the two models:
- **noalias.s:** index fetched as `s_load_b32 s0, s[4:5], 0x0` in the **prologue** (scalar
  cache, SGPR, before any branch). Zero in-loop memory ops for the index.
- **plain.s:** index fetched as `global_load_b32 v1, v0, s[4:5]` **inside `.LBB0_2`** (the hot
  loop), 8├Ś unrolled + epilogue ŌĆö a per-iteration vector load through the memory pipe.
- This is the identical mechanism the #1885 repro measured on the real kernel (a8w4 prefill:
  16 in-loop `v_readfirstlane` ŌåÆ 0; index load `global_load_u16` ŌåÆ 32├Ś `s_load_u16` in the
  prologue). The minimal model reproduces the *class* of transform; the repro's frozen asm
  is the *instance*.

### Fact 4 ŌĆö THE SILENT-MISCOMPILE mechanism (opt -O3), the load-lied case
`/tmp/td-S3/miscompile{,_plain}.ll`: read idx (#1), store 999 through `%out`, read idx (#2),
observe both. After `-O3`:
```
noalias: %a = load ...%idxp, !amdgpu.noclobber   ;   store 999, %out
         store i32 %a, %sink                       ;   store i32 %a, %sink+1   <-- read #2 ELIMINATED, reuses %a
plain  : %a = load ...%idxp                        ;   store 999, %out
         %b = load ...%idxp                         ;   store i32 %a,%sink ; store i32 %b,%sink+1  <-- both survive
```
- Under noalias, the optimizer is **licensed** to prove the store to `%out` cannot touch
  `%idxp`, so it CSEs read #2 to the pre-store value `%a`. If the caller LIED and `%out`
  overlaps `%idxp`, the store wrote a new value that the program **never observes** ŌĆö it
  keeps using the stale index. No crash, no diagnostic: a **silent miscompile**.
- Mapped to the kernel: `%a` is the gather index feeding X's row GEP. A stale index ŌåÆ the
  GEMM silently reads the WRONG token rows ŌåÆ wrong output, numerically plausible, no error.
- This is the "readonly axis" violation (a write through/into P). The "noalias axis"
  violation is the same store arriving from a *different* writable arg that secretly overlaps
  P ŌĆö same stale-read outcome. Both are the LangRef "modified during execution via a pointer
  not based on the argument" UB.
- gfx1250 layer: even the compile-time CSE aside, once the index is an `s_load` from K$, an
  in-kernel vector store into P's buffer would not invalidate the scalar cache line, so the
  scalar read is stale at the *hardware* level too ŌĆö a second, independent failure mode with
  the same silent-wrong-output symptom.

### Fact 5 ŌĆö roles, stated precisely (grounded in Facts 1ŌĆō4)
- **readonly** = "never written through THIS pointer." Establishes load *invariance/soundness
  of reuse*; it is what makes the load a candidate for hoist/`s_load` at all. Inferred by
  LLVM from the load-only use (Fact 1) ŌĆö the contract does NOT need to supply it, and toggling
  the frontend `tl.const` does not change the outcome (isolate.sh Exp A: readonly row is inert).
- **noalias** = "does not alias any OTHER (writable) arg." Establishes that no *other* store
  can clobber the location, which is the AA fact that (a) lets LICM hoist across the in-loop
  store (Fact 2) and (b) lets ISel pick scalar `s_load` (Fact 3). It is the operative switch
  (isolate.sh Exp A: noalias=1 ŌåÆ 0 in-loop, noalias=0 ŌåÆ churn, independent of readonly).
- **Cross-arg aliasing UB** = the failure when noalias is asserted but false: a store through a
  genuinely-aliasing writable arg lands in P's buffer; LLVM has already discarded the reload
  (Fact 4) and/or the K$ never sees it ŌĆö stale gather index, silently wrong output.

## Hypotheses + refutation trail

- **H1: readonly (from tl.const) is what enables s_load; noalias is secondary.**
  REFUTED. FunctionAttrs infers `readonly` with AND without noalias (Fact 1), yet only the
  noalias variant hoists to `s_load` (Facts 2ŌĆō3). isolate.sh Exp A is the independent frozen
  control: the readonly toggle is inert, the noalias toggle flips 0Ōåöchurn.
- **H2: a violation would fault or produce a detectable error.** REFUTED. The `-O3` output
  simply reuses the stale value (Fact 4) ŌĆö valid codegen, no guard. Symptom is wrong numbers.
- **H3: the risk is purely a compiler-CSE artifact (would vanish if we forced a reload).**
  PARTIALLY REFUTED / sharpened. Even forcing a reload, the gfx1250 scalar cache is not
  coherent with in-kernel vector stores absent an invalidate, so an `s_load` from K$ is stale
  at the hardware level too (Fact 4 gfx1250 layer; corroborated by the s_load/K$ path in
  widen_sload ledger). Two independent failure modes, same symptom.

## CLAIM STATUS

**CLAIM (answer):** The contract is a two-part caller promise ŌĆö (readonly) never write through
P, and (noalias) P does not overlap any writable arg. Inferred `readonly` gives load
invariance; `noalias` is the operative AA fact that yields `!amdgpu.noclobber` ŌåÆ hoisted scalar
`s_load`. Violation is UB manifesting as a silent miscompile: LLVM reuses the pre-store index
value (proven by CSE of the second load) and/or the gfx1250 scalar K$ never observes the vector
store ŌĆö either way a stale gather index ŌåÆ wrong X rows ŌåÆ silently wrong output.

**Counter-experiment that would refute it, and its result:**
- *Refuter A:* if `readonly` required `noalias`, dropping noalias would drop readonly ŌåÆ
  FunctionAttrs run shows readonly present in both (Fact 1). NOT REFUTED.
- *Refuter B:* if noalias were not the switch, toggling it would not change the s_load/in-loop
  count ŌåÆ llc shows s_load(prologue) vs global_load(in-loop) flips exactly with noalias
  (Facts 2ŌĆō3), and isolate.sh Exp A shows 0Ōåöchurn keyed to noalias only. NOT REFUTED.
- *Refuter C:* if a violation were safe, `-O3` would keep the reload across the aliasing store
  ŌåÆ it eliminates the second load under noalias (Fact 4). NOT REFUTED.

The claim is not refuted by the obvious next experiments.
