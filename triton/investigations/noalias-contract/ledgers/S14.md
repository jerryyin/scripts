## HOW-facts (real source lines + code path)

### gmm — SURVIVOR (S8 GVN/LICM alias-DCE candidate)
`_work_stealing_gmm` (`_triton_kernels/gmm.py:319`) is a persistent work-stealing loop:
```
355  while tile < total_tiles:
356      g, m, num_m_tiles, last_m, tile_in_mm = _resolve_gmm_tile(
357          group_sizes_ptr, tile, G, ...)          # <-- re-loads group_sizes EVERY iter
365      _process_gmm_tile(lhs_ptr, rhs_ptr, out_ptr, bias_ptr, ...)
390      tile = tl.atomic_add(tile_counter_ptr, 1, sem="relaxed").to(INT_TYPE)  # <-- write in loop
```
`_resolve_gmm_tile` (`:100`) and `_total_gmm_tiles` (`:81`) both do
```
112  group_sizes = tl.load(group_sizes_ptr + g_range, mask=g_mask, other=0)   # g_range = arange(0,BLOCK_SIZE_G)
```
- `group_sizes_ptr`: read-only, small (BLOCK_SIZE_G = next_pow2(G), typ. <=128), int (INT_TYPE =
  `group_sizes_ptr.type.element_ty`, int32/int64). NEVER stored to anywhere in the file.
- Address is a **vector** (`group_sizes_ptr + g_range`), so this is NOT an s_load-scalarization
  case (S5). BUT the entire `group_sizes` tile is **loop-invariant** and is re-loaded on every
  `while` iteration, and the same loop body contains `tl.atomic_add(tile_counter_ptr, ...)`.
  Under Triton's default may-alias (S1: all ptr args bare, may-alias), LLVM must assume the
  atomic store to `tile_counter_ptr` could clobber `group_sizes_ptr`'s memory → GVN/LICM cannot
  hoist the invariant `group_sizes` reload out of the loop. `noalias` on `group_sizes_ptr`
  breaks that may-alias and lets FunctionAttrs infer `readonly`, enabling the invariant hoist —
  the S8 (GVN/DSE/LICM "pessimism removed") mechanism, NOT the S5 s_load one.
- Secondary/weaker: `bias_ptr` (`:199` `bias_ptrs = bias_ptr + g*N + offs_bias_n`,
  `:200 tl.load`) is read-only but loaded once per tile with a vector offset and no interfering
  write in `_process_gmm_tile` — marginal.

### causal_conv1d — SURVIVOR (S5/T2 s_load scalarization candidate; STRONGEST)
`_causal_conv1d_fwd_kernel` (`_triton_kernels/conv/causal_conv1d.py:8`), with
`idx_seq = tl.program_id(0)` (uniform across the block, `:61`):
```
70   sequence_start_index = tl.load(query_start_loc_ptr + idx_seq)          # uniform scalar
71   sequence_end_index   = tl.load(query_start_loc_ptr + idx_seq + 1)      # uniform scalar
88   conv_state_batch_coord = tl.load(conv_state_indices_ptr + idx_seq).to(tl.int64)  # uniform scalar
111  load_init_state = tl.load(has_initial_states_ptr + idx_seq).to(tl.int1)          # uniform scalar (bool)
```
- `query_start_loc_ptr` (int32, batch+1), `cache_indices_ptr`/`conv_state_indices_ptr` (int32,
  batch), `has_initial_states_ptr` (bool, batch): all **read-only** index/flag arrays, never
  stored to. Addresses are **uniform** (offset = program_id(0), same on every lane). This is
  exactly the S5/T2 pattern: uniform + (with noalias→readonly→) invariant → ISel should pick
  scalar `s_load` instead of `v_readfirstlane` + `global_load`.
- Loads are executed once (not in a loop), so the S4/S8 LICM/GVN angle does NOT apply here — the
  win, if any, is pure scalarization (fewer VALU `v_readfirstlane`, scalar path).

  sub-dword**, which hits the sub-dword `s_load` ceiling and likely will NOT scalarize; the two
  **int32** pointers (`query_start_loc`, `cache_indices`) are the clean s_load candidates.
- `_causal_conv1d_update_kernel` (`:372`) adds more read-only index ptrs of the same shape
  (`cache_seqlens_ptr`, `conv_state_indices_ptr`, `num_accepted_tokens_ptr`) — same mechanism,

## Hypotheses + refutation trail

- **H1 "gmm group_sizes is an s_load (S5) candidate like MoE GatherIndx."** REFUTED as an S5
  case: the load address is a **vector** (`+ g_range = arange`), not uniform, so ISel cannot
  scalarize it to `s_load`. RE-CLASSIFIED, not discarded: it is a genuine **S8 GVN/LICM**
  candidate — loop-invariant read-only reload with an interfering `atomic_add` in the same loop.
- **H2 "topk index tensors (OUT_I/index_ptr) are index loads like the gather case."** REFUTED —
  they are *stored* (outputs), never read as a read-only lookup table; the may-alias that
  `noalias` removes is on the *read* side of an aux pointer, which topk lacks.
- **H3 "softmax/activation have a scalar-param pointer that could be noalias'd."** REFUTED —
  their scalar params (n_cols, scaleN, DTYPE_MAX, strides) are passed **by value**, not through a
  pointer, so there is no pointer arg to annotate.
- **H4 "causal_conv1d index loads are already scalar / nothing to gain."** NOT refuted at source
  level and NOT confirmed at asm level (compile friction, below). Left as the open question the
  full cell must answer; the sub-dword `has_initial_state` bool is pre-flagged as likely a
  non-scalarizing exception.
