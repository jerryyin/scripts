## HOW-facts (real IR/asm + exact lowering code path)

### Operand fate — where each primary operand goes (from source + backend lowering)

`gemm_a16w16` and `gemm_mxfp4` do NOT emit `tt.load`/`tt.store` on the addrspace(1) primary
operands. The three primaries take two distinct lowering paths:

1. **A/B reads → TDM DMA (both kernels), and C write → TDM (compute_bound + mxfp4):**
   `gl.amd.gfx1250.tdm.async_load` / `async_store` build a *tensor descriptor* from the base
   pointer. In `third_party/amd/lib/TritonAMDGPUToLLVM/TDMUtility.cpp:553`:
   ```
   Value globalAddr = b.ptrtoint(i64_ty, srcPtr);   // base pointer -> integer
   ...
   Value g0_3 = b.trunc(i32_ty, b.lshr(globalAddr, v32));  // packed into i32 descriptor bitfields
   ```
   The DMA is then `emitTDMRawIntrinsic` (`TDMUtility.cpp:1246`):
   ```
   const char *intrinsicName = isLoad ? "llvm.amdgcn.tensor.load.to.lds"
                                      : "llvm.amdgcn.tensor.store.from.lds";
   ```
   These intrinsics take **only i32 descriptor group operands — no pointer operand**. Once the base
   is `ptrtoint`'d, pointer provenance (and any `noalias` arg attr) is gone; LLVM AA sees an opaque
   integer-argument call.

2. **C write → buffer_store (a16w16 bandwidth_bound only):** `gl.amd.gfx1250.buffer_store`
   (gemm_a16w16.py:379) lowers via `BufferOpsEmitter.cpp:107`:
   ```
   Value resource = createOrFold<ROCDL::MakeBufferRsrcOp>(loc, ptr_addrspace8, basePtr, ...);
   ```
   i.e. `llvm.amdgcn.make.buffer.rsrc(basePtr=c_ptr, ...) -> ptr addrspace(8)`, then
   `ROCDL::RawPtrBufferStoreOp` (`BufferOpsEmitter.cpp:229`). Here the C base pointer stays a real
   pointer operand of `make.buffer.rsrc`, so its `noalias` attr CAN reach AA (Fact 1).

3. **bias_ptr → plain `gl.load`** (read-only, one load before the store). Not a primary A/B/C; noted
   for completeness.

### Fact 1 — noalias PROPAGATES through make.buffer.rsrc; GVN forwards across the C buffer store
`/tmp/s11/buf_rsrc.ll`: `load q; make.buffer.rsrc(c); buffer.store(c); load q` — A/not-A on `%c`.
`opt -passes=gvn` (`/tmp/s11/buf_rsrc_gvn.ll`):
```
@rmw_mayalias (ptr addrspace(1) %c)          : loads=2   ; store-to-C may clobber q -> NOT forwarded
@rmw_noalias  (ptr addrspace(1) noalias %c)  : loads=1   ; C !alias Q -> 2nd load forwarded/deleted
```
So the buffer-op lowering does NOT sever noalias: `noalias` on the addrspace(1) base survives
`make.buffer.rsrc` into an alias-gated middle-end opt on the *written* operand. The mechanism CAN
reach a primary WRITE operand.

### Fact 2 — noalias is INERT on the TDM (A/B) path (ptrtoint destroys provenance)
`/tmp/s11/tdm_ptrtoint.ll`: base `ptrtoint`'d and fed to an opaque integer-arg DMA stub (faithful
model of `tensor.load.to.lds`), A/not-A on the base. After `opt -passes=gvn,dse,instcombine`
(`/tmp/s11/tdm_out.ll`) the two function BODIES are byte-identical (only the signature `noalias`
token differs):
```
BODIES_IDENTICAL: noalias inert once base is ptrtoint'd into TDM descriptor
```
noalias on an integer-consumed base cannot inform AA about any DMA — there is no pointer memory op.

### Fact 3 — realistic F3 epilogue: single trailing C store — noalias changes NOTHING in asm
`/tmp/s11/c_store_only.ll` models the actual gemm_a16w16 bandwidth_bound epilogue: one
`make.buffer.rsrc(c)` + one `raw.ptr.buffer.store.v4f32` as the last op, no reload. A/not-A on `%c`.
`llc -mcpu=gfx1250 -O3` (`/tmp/s11/c_store_only.s`): both functions emit the identical
```
s_load_b128 ... ; s_load_b32 ... ; s_load_b64 ... ; buffer_store_b128 v[0:3], v4, s[0:3], null offen
```
asm bodies byte-identical (only the label comment `; @cstore_plain` vs `; @cstore_noalias` differs).
There is no reload from C and no other global op after the store, so the alias promise of Fact 1 has
no construct to act on.

### Fact 4 — firing-site enumeration in the real F3 kernels (source, definitive)
Global-memory-op order (grep of the two F3 kernels):
- **a16w16 bandwidth_bound:** A/B = TDM `async_load` (lines 178–298); `bias_ptr` = one plain
  `gl.load` (360, read-only); C = single `buffer_store` (379) as the LAST global op. No reload from
  C, no in-loop global store, no interleaved global RMW. => no GVN/DSE/CSE/reorder firing site
  (matches S8's structural prediction verbatim).
- **a16w16 compute_bound:** A/B = TDM `async_load`; C = LDS `c_buffer.store` then TDM
  `async_store` (827). Every primary is on the TDM path -> Fact 2 (noalias inert). No global RMW.
- **mxfp4:** A/B/scales = TDM `async_load` (289–331); C = `c_buffer.store` + TDM `async_store`
  (395–397). All primaries on TDM path -> Fact 2. No global RMW.

### Fact 5 — plumbing (S2 angle): primary annotation IS expressible today; only noalias is emitted
- `python/triton/runtime/jit.py:817` — `na = i in noalias_args or param.name in noalias_args`:
  `noalias_args` accepts arg **names OR indices**, so `noalias_args=["a_ptr","b_ptr","c_ptr"]` on
  the F3 `@gluon.jit` is fully expressible; kernel arg names confirmed to be `a_ptr,b_ptr,c_ptr,
  bias_ptr`. **No plumbing gap blocks annotating primary operands for F3.**
- `jit.py:731` gates it to real (non-constexpr) pointer args -> `["tt.noalias",1]` arg attr.
- Sink `lib/Conversion/TritonGPUToLLVM/Utility.cpp:2233 handlePointerContractArgs`: for each
  `tt.func` PointerType arg with `tt.noalias`, sets `llvm.noalias` ONLY. It never sets
  `readonly`/`writeonly` (grep: zero readonly/writeonly plumbing from the frontend). This is
  correct for the WRITE operand C: `noalias` is a pure aliasing promise, orthogonal to read/write,
  so annotating a written pointer is sound. `readonly` is separately *inferred* by LLVM
  FunctionAttrs from use (S8 Fact A) and will NOT be inferred for C (it is stored to) — which is
  exactly right and not a blocker.

## Hypotheses + refutation trail

- **H1 "noalias on A/B enables buffer_load / vectorization / scheduling on the read operands."**
  REFUTED — A/B never become addrspace(1) loads; they are TDM DMAs whose base is ptrtoint'd
  (operand fate #1), and noalias on a ptrtoint-consumed base is inert (Fact 2). (Consistent with
  S6: buffer selection is offset-range-gated, not alias-gated, and A/B aren't even buffer/global
  loads here — they are TDM.)
- **H2 "noalias on C (written) is unexpressible / blocked by its read-write nature."** REFUTED —
  annotation is expressible (Fact 5), the attr propagates through make.buffer.rsrc to the buffer
  store (Fact 1), and the write nature does not block noalias (only `readonly` would be wrong for a
  written ptr, and the sink never emits `readonly`). The block is the *absence of a firing site*,
  not the read/write nature.
- **H3 "noalias on C unblocks a real opt in gemm_a16w16."** REFUTED for the kernel as written —
  Fact 3 (single trailing store: identical asm) + Fact 4 (no reload/RMW/interleave). The Fact-1
  mechanism is real but dormant: no matching construct. (It WOULD fire in a hypothetical fused
  in-place / read-modify-write-C epilogue — see unknowns.)
- **H4 "the read/write nature fundamentally blocks noalias."** REFUTED as the cause — the true
  fundamental blockers are (a) TDM `ptrtoint` erasing provenance for A/B (and TDM-path C), and
  (b) no alias-gated construct for the buffer_store C. Read/write only matters in that `readonly`
  is (correctly) not inferred for the written C — which is not a blocker for `noalias` itself.

## CLAIM STATUS

**CLAIM (answered, irrefutable at IR + asm + lowering-code + plumbing level):**
For the F3 kernels **as written**, noalias on primary A/B/C operands unblocks **nothing**:
- **A/B (read):** on the TDM path; base `ptrtoint`'d into an integer descriptor, DMA intrinsic has
  no pointer operand -> noalias INERT (Fact 2, operand fate #1). Neither necessary nor sufficient —
  simply unreachable.
- **C (write):** two sub-cases.
  - bandwidth_bound: buffer_store; noalias PROPAGATES through make.buffer.rsrc and IS sufficient to
    unblock GVN load-forwarding *in principle* (Fact 1) — but there is NO firing site (single
    trailing store; Fact 3/4), so the observed effect is zero.
  - compute_bound / mxfp4: TDM async_store; base ptrtoint'd -> noalias INERT (Fact 2).
- **buffer_load / vectorization / scheduling:** none unblocked here — buffer selection is
  range-gated not alias-gated (S6); widening needs adjacent same-base global loads/stores which the
  TDM/single-store structure does not present (S7 mechanism has no site); classic middle-end opts
  have no firing site (S8, re-confirmed by Fact 4).
- **Fundamental blockers:** (1) TDM ptrtoint provenance loss (A/B and TDM-path C); (2) absence of
  an alias-gated construct (no in-loop/epilogue global RMW to C) for the buffer_store path. The
  read/write nature is NOT the blocker; the plumbing is NOT the blocker (primary annotation is
  expressible today, Fact 5).

**Counter-experiments that would refute (and their results):**
- *"Show noalias on the TDM base changes downstream IR."* -> Fact 2: bodies byte-identical.
  Not refuted.
- *"Show noalias on C fails to reach AA through the buffer-op lowering."* -> Fact 1: it reaches AA
  (GVN 2->1). So the mechanism is NOT severed — the reason for no win is the missing site, proven
  separately (Fact 3). Not refuted.
- *"Show the real gemm_a16w16 C store asm differs with noalias."* -> Fact 3: identical asm body.
  Not refuted.
- *"Show primary annotation is not expressible (plumbing gap)."* -> Fact 5: `noalias_args` accepts
  arg names incl. `a_ptr/b_ptr/c_ptr`. Not refuted.
