## HOW-facts (evidence — real IR/asm)

### Fact A — the entire IR delta is a single `noalias` token
Full `diff` of the two post-Triton-opt `.llir` (a8w4 prefill) = one changed line, the `define`:
```
< ... i32 %18, ptr addrspace(1)         nofree readonly captures(none) %19, ...
> ... i32 %18, ptr addrspace(1) noalias nofree readonly captures(none) %19, ...
```
(and the analogous 1-line diff in the pre-opt pair). `readonly captures(none)` is present on
%19 in **both** — it is inferred from the use pattern (load-only), NOT derived from `noalias`.
So S1's "noalias⇒readonly" shorthand is, for *this* pointer, use-based inference; `noalias` adds
only the alias promise. All alias-gated behavior below is attributable to that one token.

### Fact B — GVN / DSE / CSE / MemCpyOpt / NewGVN produce IDENTICAL IR (isolated passes)
On the pre-opt pair, each pass run alone (`opt -passes=<p> -mcpu=gfx1250 -S`) yields output whose
only diff is the ModuleID comment + the define-line `noalias` token — zero instruction-level
difference:
```
gvn        : non-define diffs = (none)
newgvn     : non-define diffs = (none)
dse        : non-define diffs = (none)
memcpyopt  : non-define diffs = (none)
```
### Fact C — full -O3 middle-end stats are IDENTICAL before/after
`opt -O3 -mcpu=gfx1250 -stats` line-diff before vs after = empty. Notably identical:
`aa: 853 MayAlias / 5365 NoAlias` (unchanged — noalias adds no NoAlias results the analysis
didn't already have), `gvn: 1098 deleted / 1060 simplified`, `dse: 32 stores remaining`,
`early-cse: 109 CSE'd / 46 GEP CSE'd`, `licm: 8 hoisted`, `SLP: 448`. And the two `.llir`
have identical counts: **325 loads, 32 stores, 0 calls** in both.
=> the Triton middle-end (which runs GVN/CSE/DSE/MemCpyOpt) did the exact same work with and
without noalias.

### Fact D — WHY the middle-end opts cannot fire here (mechanism, traced)
In `_moe_gemm_a8w4_prefill` pre-opt:
- Store address-space histogram: **32 stores, ALL `store ..., ptr addrspace(3)`** (LDS). **Zero
  `store ..., ptr addrspace(1)`** (no global store anywhere in the function).
- GatherIndx feeds `%190 = getelementptr i16, ptr addrspace(1) %19, ...` then
  `%223 = load <1 x i16>, ptr addrspace(1) %191` (37 such global index loads, in the entry
  block, before the hot loop).
- LLVM AA proves `addrspace(1)` vs `addrspace(3)` **NoAlias by construction** — no attribute
  needed. So the only candidate for GVN-load-forwarding / DSE / MemCpyOpt (a global load that
  might be clobbered by a store) has **no clobbering store to be gated on**. `noalias` on %19
  removes a may-alias edge that basicaa had already removed. There is literally nothing for the
  alias-gated middle-end opts to do differently. (F3 GEMM has the same shape from source:
  A/B global loads -> TDM async into LDS -> WMMA in LDS -> a single C global store at the end;
  loads and the store never interleave, so no in-loop load-across-store candidate either.)

### Fact E — the real win is a BACKEND ISel effect, not a middle-end opt
Same-IR-into-stock-llc, whole-function asm histogram:
```
BEFORE (may-alias): s_load= 9  global_load=37  v_readfirstlane=37
AFTER  (noalias)  : s_load=41  global_load= 5  v_readfirstlane= 3
```
Hot-loop body (the reproducer's `report()` range), instruction categories:
```
BEFORE: 66 ds_ , 16 v_readfirstlane
AFTER : 66 ds_ ,  0 v_readfirstlane
```
The hot-loop delta is exactly **-16 `v_readfirstlane`**; the 66 LDS (`ds_`) ops are byte-for-byte
unchanged. noalias makes the GatherIndx address/load uniform+invariant, so AMDGPU ISel selects
scalar `s_load` and drops the per-iteration `v_readfirstlane` broadcast. This is T2 territory
(scalarization), NOT GVN/DSE/CSE/MemCpyOpt.

### Fact F — no alias-driven scheduling reorder; only scalar-math reshuffle
Hot-loop opcode sequence (registers stripped), BEFORE-minus-rfl vs AFTER: the memory-op
(`ds_*`) ordering is identical; the only reordering is among scalar address-math
(`s_mul_hi_u32`/`s_mul_i32`/`s_sub_co_i32`/`s_max_i32`) — a byproduct of the address computation
moving from VALU+readfirstlane to pure SALU, not an alias-relaxed load/store schedule. So
"instruction scheduling freedom" in the alias sense is NOT unblocked here.

### Fact G — noalias IS sufficient to unblock GVN load-forwarding *in the abstract*
`/tmp/tc-S8/synth.ll`: `load p; store q; load p` in two functions.
```
opt -passes=gvn : @t_mayalias -> 2 loads (store may clobber -> no forward)
                  @t_noalias  -> 1 load  (noalias p != q -> forward, 2nd load deleted)
```
So the mechanism is real and noalias is sufficient to fire GVN — it simply has no matching
construct in the AITer kernels (Fact D). This is the honest theoretical-vs-observed split.

### Fact H — correctness (FFM, gfx1250, reduced prefill size batch=2048/block_m=128)
`run_moe_gemm_ffm.py --kernel a8w4 --backend gluon --phase prefill`, both configs:
```
noalias OFF: forward: PASS  rel_err=0.01001  cosine=0.999927   RESULT: PASS
noalias ON : forward: PASS  rel_err=0.01001  cosine=0.999927   RESULT: PASS
```
Identical numerics (rel_err is the a8w4 quant-vs-dequant tolerance, not a noalias artifact) —
the contract is a pure hint with no correctness change, consistent with there being no
alias-gated transform to (mis)fire.

## Hypotheses + refutation trail

- **H1 "noalias unblocks GVN/CSE redundant-load elimination here."** REFUTED for F1 — Fact B
  (isolated GVN/NewGVN identical), Fact C (gvn stats identical), Fact D (no clobbering global
  store exists; global-vs-LDS already NoAlias). Confirmed *possible in principle* by Fact G, so
  the honest verdict is "sufficient in general, not observed here."
- **H2 "noalias unblocks DSE."** REFUTED — Fact B/C (dse identical, `32 stores remaining` both);
  Fact D (the 32 stores are LDS, dead-store analysis is unaffected by a global pointer's alias
  set). noalias on %19 cannot make any store dead.
- **H3 "noalias unblocks MemCpyOpt."** REFUTED — Fact B (memcpyopt identical output); no
  memcpy/memset idiom on the GatherIndx path.
- **H4 "noalias gives the scheduler freedom to reorder loads past stores."** REFUTED — Fact F
  (ds_ memory ordering unchanged; only scalar-math shuffle from scalarization). The
  load(global)/store(LDS) pair was already reorderable (different addrspace), so noalias grants
  no new scheduling latitude.
- **H5 "the observed win proves a classic alias opt fired."** REFUTED — Fact E localizes the
  entire win to ISel scalar `s_load` selection + dropped `v_readfirstlane` (T2), with identical
  middle-end IR (Fact C) — a codegen effect, categorically not GVN/DSE/CSE/MemCpyOpt.

**One-line answer:** For these AITer gfx1250 kernels, `noalias` unblocks **none** of GVN/CSE,
DSE, MemCpyOpt, or alias-gated scheduling — those are theoretically enable-able by noalias
(GVN proven synthetically) but have no firing site because all stores are LDS (addrspace 3),
already NoAlias to the addrspace(1) gather loads, and global load/store never interleave. The
real, observed win is entirely a backend ISel scalarization (s_load, -16 in-loop
v_readfirstlane), which is not one of the S8 classic opts.

## Remaining unknowns (hand-offs, not blockers)

- **F3-primary (S11):** the genuine may-alias case for classic opts is a kernel that both reads
  and writes global memory *with interleaving* (e.g. a fused/in-place epilogue, or a
  read-modify-write to C). gemm_a16w16 as written (load->LDS->WMMA->single trailing C store)
  does not interleave, so it inherits F1's "no firing site" result. If S11 finds/needs a kernel
  with an in-loop global store between global loads, re-run the Fact-B/G methodology there — that
  is the only place GVN/DSE could plausibly be *observed*. Not investigated here (no build).
- **The `s_load`/scalarization win itself** is S5/T2's subject; S8 only established it is NOT a
  classic middle-end alias opt.
- Whether emitting `!alias.scope`/`!noalias` metadata on individual loads (vs the arg attr)
  could reach opts the arg attr doesn't is a separate lever, not exercised here.
