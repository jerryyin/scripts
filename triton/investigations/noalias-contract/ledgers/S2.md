## The full plumbing (traced, def-to-use, real citations)

### Stage 1 ‚Äî Frontend decorator surface (`python/triton/runtime/jit.py`)
- `jit(..., noalias_args=None)` (jit.py:797). `noalias_args or []` (jit.py:800),
  stored as `self.noalias_args` (jit.py:807). **The `self.noalias_args = ...` storage
  is the #1885 fix** ‚Äî the commit message: *"Fix noalias_args propagation: store
  self.noalias_args so triton_kernels.specialize carries it into activation-fused
  kernels (e.g. `_matmul` -> `_matmul_swiglu_fn`), which otherwise silently drop the
  contract."* Before the fix the kwarg was consumed but not persisted, so
  `specialize()`-derived kernels lost it.
- Per-param resolution (jit.py:817): `na = i in noalias_args or param.name in
  noalias_args` ‚Äî accepts **index OR name**, no restriction on which arg.
- `KernelParam.__init__(..., noalias: bool = False)` (jit.py:307) ‚Äî **default False**;
  stored `self.noalias = noalias` (jit.py:314).

### Stage 2 ‚Äî Gluon mirror (`python/triton/experimental/gluon/_runtime.py`)
- `gluon.jit(..., noalias_args=None)` at lines 71 & 86 (both overload + impl), passed
  straight through to `GluonJITFunction(..., noalias_args=noalias_args)` (line 117).
  Gluon adds no logic of its own ‚Äî it inherits the jit.py machinery. So the gluon and
  triton paths are identical past this point.

### Stage 3 ‚Äî Attr emission at compile time (`jit.py` `_get_options`, ~line 726)
```python
for p in self.params:
    if p.noalias and specialization[p.num][0] != "constexpr":
        attrs.setdefault((p.num,), []).append(["tt.noalias", 1])
```
- Emits `tt.noalias` **only** for an opted-in param that is a **real (non-constexpr)
  runtime pointer**. The `!= "constexpr"` guard is the #120 fix for the `do_gather=False`
  IndexError (an optional ptr passed `None` specializes to constexpr ‚Üí not a runtime
  arg ‚Üí tagging it is out-of-range). This is the **only** arg-eligibility gate in the
  whole chain ‚Äî and it is about constexpr-ness, **not** arg role.

### Stage 4 ‚Äî Attr ‚Üí `tt.func` arg attribute (`python/triton/compiler/code_generator.py:275`)
```python
attr_specs = self.attrs.get(path, [])
for attr_name, attr_val in attr_specs:
    fn.set_arg_attr(arg_idx, attr_name, attr_val)     # line 275
```
- Fully generic: whatever `["tt.noalias",1]` is in the attrs dict for a path is applied
  to that arg. No role check.

### Stage 5 ‚Äî Backend sink, the ONLY place a func-arg alias attr is set
`third_party/amd/lib/TritonAMDGPUToLLVM/FuncOpToLLVM.cpp`:
- `FuncOpConversion::matchAndRewrite` calls (line 29) `handlePointerContractArgs(funcOp,
  newFuncOp)` right after `handleArgPtrDatatype` (line 28).
`lib/Conversion/TritonGPUToLLVM/Utility.cpp:2233` (current, post-#120):
```cpp
void handlePointerContractArgs(triton::FuncOp funcOp, LLVM::LLVMFuncOp &llvmFuncOp) {
  // Map the tt.noalias caller contract to LLVM noalias on pointer args. ...
  for (unsigned i = 0; i < fty.getNumInputs(); ++i) {
    if (!isa<triton::PointerType>(fty.getInput(i))) continue;
    if (funcOp.getArgAttr(i, "tt.noalias"))
      llvmFuncOp.setArgAttr(i, LLVM::LLVMDialect::getNoAliasAttrName(),
                            UnitAttr::get(ctx));
  }
}
```
- Loops **all** pointer inputs, no arg-role/index restriction. `tt.noalias` is a plain
  **string-keyed** MLIR attr ‚Äî NOT a registered TritonAMDGPU dialect attr
  (`Dialect.cpp` has zero references to it; grep clean). So the contract has no dialect
  surface ‚Äî it is a bare attr key + one setArgAttr.

### Stage 6 ‚Äî What LLVM param attrs the contract can emit today
`grep 'getReadonlyAttrName|getWriteOnlyAttrName|getNoAliasAttrName|getReadnoneAttrName'`
across `lib/` + `third_party/amd/lib/` = **one hit**: the `getNoAliasAttrName` above.
**The contract emits ONLY `llvm.noalias` ‚Äî no readonly, writeonly, align,
dereferenceable, nocapture path exists anywhere.** (Consistent with S1 Fact 1/2.)

### Fact C ‚Äî the deletion that removed readonly (git proof)
`git show ba4fd67b8e -- lib/Conversion/TritonGPUToLLVM/Utility.cpp` removed:

  user *declare* readonly on an arbitrary (non-const) pointer.**

## Answers to the four sub-questions

**(a) What attribute(s) can the contract currently emit, on which args?**
- Exactly **one**: `llvm.noalias`. On **any** pointer arg the frontend tags with
  `tt.noalias` via `noalias_args` (by index or name), with the single eligibility rule
  that the arg is a real non-constexpr pointer at that call. No other LLVM param attr
  (readonly/writeonly/align/deref/nocapture) is emitted by the contract path.

**(b) Can primary operands be annotated today, or what blocks it?**
- **Yes, today, with no backend change** (Fact A). `handlePointerContractArgs` has no
  arg-role gate; A/B and even the write operand C all receive `llvm.noalias`. The
  reason only `GatherIndx` is annotated in-tree is purely that only MoE `moe_op_gemm_a8w4.py`
  lists `noalias_args` (S1 Fact 6). To annotate a GEMM's A/B/C you only add
  `noalias_args=["A","B","C"]` to that kernel's `@gluon.jit`. *Soundness* of doing so
  (is C-write-vs-A/B-read actually non-aliasing? cross-arg UB?) is **S3/T6's** job, not
  a plumbing blocker; the mechanism permits it.

**(c) Can readonly be user-declared, or only LLVM-inferred?**
- **Only LLVM-inferred today** (Facts B & C). There is no `readonly_args` kwarg and no
  live `tt.readonly‚Üíllvm.readonly` mapping ‚Äî both the frontend derivation (from const)
  and the backend mapping were deleted in #120. `tt.readonly` is inert metadata. The
  only way an arg becomes `readonly` is LLVM's `FunctionAttrs` inferring it post-opt
  (which #120 relies on ‚Äî see S1 remaining-unknowns / S4).

**(d) Minimal surface change to annotate arbitrary args (+ expose readonly).**
- **Annotate arbitrary args (incl. primary): ZERO code change** ‚Äî already works; it is a
  per-kernel frontend annotation (`noalias_args=[...]`). No plumbing edit needed.
- **Re-expose user-declared readonly:** minimal change is symmetric to the deleted code:
  1. Frontend: a `readonly_args` kwarg on `jit`/`gluon.jit` mirroring `noalias_args`
     (KernelParam.readonly, emit `["tt.readonly",1]` in `_get_options` with the same
     non-constexpr guard). ~ the mirror of jit.py:307/314/726 & 797/807/817.
  2. Backend: re-add the ~3-line `if getArgAttr(i,"tt.readonly")
     setArgAttr(getReadonlyAttrName())` block in `handlePointerContractArgs`
     (revert of Fact C's deletion). One `LLVM::LLVMDialect::getReadonlyAttrName()` call.
  - (Optional, for write operands: a `writeonly_args`/`getWriteOnlyAttrName` twin ‚Äî same
    shape.) No dialect registration is needed since the attrs are string-keyed.

## Remaining unknowns (hand-offs, not blockers)

- **Soundness of primary-operand noalias (‚Üí S3/T6):** plumbing PERMITS tagging C (write)
  as noalias, but whether that is *correct* (C must not alias A/B; cross-arg aliasing is
  caller-promised UB) is T6. S2 only proves the mechanism does not block it.
- **Whether re-exposing readonly buys anything post-#120 (‚Üí S4/T1):** #120's premise is
  LLVM FunctionAttrs already infers readonly for free. A user `readonly_args` would only
  matter where inference fails (e.g. a pointer that is read-only in fact but where LLVM
  can't prove no-store, or where a store is control-dependent). Quantifying that gap is
  mechanism work (T1/T5), not plumbing.
- **align/dereferenceable from tt.divisibility/pointer_range:** still not emitted (S1
  Fact 1); adding them is the same one-line-per-attr shape but is a separate T2/T3 ask.
