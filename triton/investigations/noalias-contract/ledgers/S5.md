CONCLUSION: On gfx1250, ISel selects a scalar SMEM `s_load` for a load IFF five conditions all hold — (1) address uniform/non-divergent (or MMO uniform), (2) provably no-clobber [MMO `invariant` from readonly OR `MONoClobber` from noalias], (3) global/constant addrspace with scalarize-global on, (4) natural alignment >= 4, and (5) the load is either dword-granular (i32/i64: coalesces to wide `s_load_dwordx{1,2,4,8,16}`) OR a scalar sub-dword *extload-to-i32* (per-element `s_load_u8/u16`, one SGPR each); the sub-dword ceiling is that AMDGPU has NO sub-dword *vector* SMEM instruction, so any contiguous sub-dword load that the vectorizer coalesces into a `<N x i16>`/`<N x i8>` node has no scalar form and falls to VMEM `global_load`/`flat_load` even when uniform+noclobber.

## HOW-facts — the ISel condition set, each proven on real asm/IR

### FACT A — the five conditions (LLVM source, exact lines)
Two decision sites in `SIISelLowering.cpp`:

1. **Sub-dword SMEM legality** — `isLegalAddressingMode`, lines 2024-2031:
```
    if (!Subtarget->hasScalarSubwordLoads()) {
      // There are no SMRD extloads, so if we have to do a small type access we
      // will use a MUBUF load.
      if (Ty->isSized() && DL.getTypeStoreSize(Ty) < 4)
        return isLegalGlobalAddressingMode(AM);   // <-- forces VMEM for sub-dword
    }
```
   `GCNSubtarget.h:248  bool hasScalarSubwordLoads() const { return getGeneration() >= GFX12; }`.
   gfx1250 is GFX12+ => `hasScalarSubwordLoads()==true`, so this branch does NOT bite: scalar
   sub-dword SMEM (`s_load_u8/u16`) IS available on gfx1250. (This corrects the naive "gfx has no
   sub-dword s_load" — it has the *scalar* one; what it lacks is the *vector* one, Fact C.)

2. **Uniform-invariant scalarization gate** — the vector-load legalizer, lines 13148-13158:
```
  if (AS == CONSTANT_ADDRESS || CONSTANT_ADDRESS_32BIT ||
      (AS == GLOBAL_ADDRESS && Subtarget->getScalarizeGlobalBehavior() &&
       Load->isSimple() && (Load->isInvariant() || isMemOpHasNoClobberedMemOperand(Load)))) {
    if ((!Op->isDivergent() || AMDGPU::isUniformMMO(MMO)) &&
        Alignment >= Align(4) && NumElements < 32) { ...scalar SMEM path... }
```
   `isMemOpHasNoClobberedMemOperand` (line 2335) = `MMO->getFlags() & MONoClobber` — i.e. the
   `!amdgpu.noclobber` flag S4 traced to `AMDGPUAnnotateUniformValues` (AA-derived from `noalias`).
   So the gate literally reads: **uniform** (`!isDivergent||isUniformMMO`) AND **noclobber-or-
   invariant** AND **align>=4** AND scalarize-global-on AND global/const addrspace.

   The five conditions: (1) uniform, (2) invariant/noclobber, (3) addrspace+scalarize,
   (4) align>=4, (5) representable as a scalar SMEM node (dword-multiple, or scalar sub-dword
   extload) — Fact C.

### FACT B — the sub-dword *vector* has no SMEM form (SMInstructions.td, structural)
The only sub-dword scalar loads defined:
```
329: defm S_LOAD_I8   : SM_Pseudo_Loads <SReg_64, SReg_32_XM0_XEXEC>;
330: defm S_LOAD_U8   : SM_Pseudo_Loads <SReg_64, SReg_32_XM0_XEXEC>;
331: defm S_LOAD_I16  : SM_Pseudo_Loads <SReg_64, SReg_32_XM0_XEXEC>;
332: defm S_LOAD_U16  : SM_Pseudo_Loads <SReg_64, SReg_32_XM0_XEXEC>;
```
Result register class is **`SReg_32`** (one 32-bit SGPR); the match patterns (1044-1049) are all
`smrd_{s,z,}extloadi{8,16} -> i32` — i.e. sub-dword SMEM only exists as a **scalar extload to a
single i32 SGPR**. All wide SMEM loads (`S_LOAD_B32/B64/B128/B256/B512`) are dword-multiples.
Grep for any `S_LOAD_(U|I)(8|16)` writing `SReg_64/96/128/...` => NONE. There is no
`s_load_u16x4`. => a sub-dword *vector* is structurally unrepresentable in SMEM.

### FACT C — the ceiling, in the DAG: contiguous sub-dword coalesces to a vector => VMEM
`/tmp/s5/matrix2.ll` (all-live xor-reduced, inline in the kernel so uniformity holds).
`@i16c8` (8 contiguous i16, `noalias readonly`, `!invariant.load`), ISel DAG node:
```
t20: v8i16,ch = load<(invariant "amdgpu-noclobber" load (s128) ... align 2 ... addrspace 1)>
```
The load is `v8i16` and IS flagged `invariant "amdgpu-noclobber"` (uniformity+noclobber legs BOTH
satisfied) — yet selects VMEM:
```
@i16c8  -> flat_load_ushort x8   (VMEM)
@i8c8   -> flat_load_ubyte  x8   (VMEM)
@i32c8  -> s_load_dwordx8         (wide SMEM)   [8xi32 coalesced]
@i64c4  -> s_load_dwordx8         (wide SMEM)   [4xi64 coalesced]
```
Contrast `@t16nc` (scattered i16), DAG keeps 4 **scalar** `i16 load (s16)` nodes (no vector
coalesce), each morphs to `S_LOAD_U16_IMM ... : i32` (SMEM). So the flip is purely scalar-vs-
vector shape: LoadStoreVectorizer coalesces the *contiguous* sub-dword loads into a `v8i16`
node, which has no SMEM instruction (Fact B) => VMEM. The scattered ones can't coalesce => stay
scalar => scalar sub-dword SMEM. **This is the sub-dword ceiling: width>=dword to be a wide SMEM
vector; sub-dword can be SMEM only per-element (scalar extload-to-i32).**

### FACT D — the alignment boundary (align>=4), isolated
`/tmp/s5/align.ll`, single i16 load `zext` to i32, `noalias readonly !invariant.load`, only the
alignment differs:
```
@one0_a4 (align 4) -> s_load_dword s0, s[0:1], 0x0   (SMEM; DAG widens the aligned subword to dword)
@one0_a2 (align 2) -> flat_load_ushort v2, v[0:1]     (VMEM)
```
Direct A/not-A: align>=4 is a hard SMEM requirement (gate line 13154 `Alignment >= Align(4)`).
(a8w4's index base is `tt.divisibility=16` => align satisfied; and its per-element extload path
uses the scalar `S_LOAD_U16_IMM->i32` form, Fact E.)

### FACT E — the invariance/noclobber leg, isolated (S4's `!amdgpu.noclobber` at ISel grain)
`/tmp/s5/invariance.ll`, two uniform i32 loads from `%p` with a store to may-alias `%o` between:
```
@i32_mayalias      (%p plain, may-alias store): load1 -> s_load_dword (pre-store, noclobber)
                                                 load2 -> flat_load_dword (clobbered => VMEM)
@i32_noalias_store (%p noalias, same store):     BOTH  -> s_load_dword (noalias => store cannot
                                                          clobber => both noclobber => both SMEM)
```
The ONLY delta is `noalias` on `%p`. This is the ISel-grain kill switch for the noclobber leg,
matching S4's `AA(noalias)=NoAlias => !isClobberedInFunction => !amdgpu.noclobber`.
Corollary (`/tmp/s5/uniformity.ll` `@i32_noinv_noalias`, `readonly` only, no `!invariant.load`,
no `noalias`): still `s_load_dword` — `readonly` alone makes the MMO invariant, so the noclobber-
or-invariant leg is satisfiable by `readonly` too. Consistent with S4 (readonly inferred in both
states; the win tracks `noalias` only where a *store* can otherwise clobber).

### FACT F — the uniformity leg, isolated
`/tmp/s5/uniformity.ll`, address made divergent (`getelementptr ptr, tid`):
```
@i16_divergent -> flat_load_ushort (VMEM)   [uniformity fails, sub-dword]
@i32_divergent -> flat_load_dword  (VMEM)   [uniformity fails, even for dword]
```
Divergent address => VMEM regardless of dtype/noalias/invariant (gate line 13154
`!Op->isDivergent() || isUniformMMO` fails). Uniformity is a separate, noalias-independent
precondition (S4 Hop 4 same finding).

## THE TRUTH TABLE (real asm, gfx1250, both llc pins agree)

| case | dtype | addr shape | uniform | noclobber/inv | align | ISel result | SMEM? |
|------|-------|-----------|---------|---------------|-------|-------------|-------|
| i32c8 | i32 (dword) | contiguous | yes | yes | 4 | `s_load_dwordx8` | wide SMEM |
| i64c4 | i64 (dword) | contiguous | yes | yes | 8 | `s_load_dwordx8` | wide SMEM |
| t32 | i32 | contiguous | yes | yes | 4 | `s_load_dword(x4)` | wide SMEM |
| t16nc | i16 (subdword) | scattered | yes | yes | 2 | `s_load_u16` x4 (->i32) | scalar SMEM |
| a8w4 (S4) | i16 (subdword) | scattered(%M) | yes | yes(noalias) | 16 | `s_load_u16` x32 (->i32) | scalar SMEM |
| i16c8 / t16 | i16 (subdword) | contiguous | yes | yes | 2 | `flat/global_load_ushort` | **VMEM** (no subdword vec SMEM) |
| i8c8 | i8 (subdword) | contiguous | yes | yes | 1 | `flat_load_ubyte` | **VMEM** (same ceiling) |
| one0_a2 | i16 | single | yes | yes | 2 | `flat_load_ushort` | **VMEM** (align<4, native i16) |
| one0_a4 | i16 | single | yes | yes | 4 | `s_load_dword` | SMEM (aligned=>widened) |
| i32_mayalias L2 | i32 | uniform | yes | **NO** (clobbered) | 4 | `flat_load_dword` | **VMEM** |
| i16_divergent | i16 | divergent | **NO** | yes | 2 | `flat_load_ushort` | **VMEM** |
| i32_divergent | i32 | divergent | **NO** | yes | 4 | `flat_load_dword` | **VMEM** |

**s_load fires IFF:** uniform-addr AND (invariant OR noclobber) AND align>=4 AND global/const AND
scalarize-global-on AND [dword-multiple => wide SMEM] OR [sub-dword scalar extload-to-i32 =>
per-element `s_load_u8/u16`]. **s_load does NOT fire when:** address divergent (Fact F); a may-
alias store clobbers and no noalias (Fact E); align<4 for a native sub-dword (Fact D); OR the load
is a **sub-dword vector** (contiguous sub-dword coalesced by the vectorizer — Fact C, the ceiling).

## Hypotheses + refutation trail

- **H1 "gfx1250 has no sub-dword s_load, so a8w4's u16 index can never be SMEM."** REFUTED —
  `hasScalarSubwordLoads()=GFX12+`=true (Fact A); scattered u16 => `s_load_u16` (Fact C, t16nc).
  The correct statement: no sub-dword *vector* SMEM (Fact B).
- **H2 "the ceiling is total-bytes<4."** REFUTED — i32c8 (32 bytes) and i64c4 (32 bytes) are wide
  SMEM; the ceiling is per-*element* width<4 combined with vector coalescing, not total size.
- **H3 "contiguity itself forces VMEM."** REFUTED (and this corrects widen_sload iteration 6's
  interim wording) — contiguous i32/i64 go wide SMEM (Fact C). Contiguity forces VMEM *only* for
  sub-dword, because contiguity triggers vectorization into a subword vector that has no SMEM form.
- **H4 "extload-to-i32 alone makes a sub-dword load SMEM."** REFUTED — `/tmp/s5/exthyp.ll`
  `@i16_zext` (single i16 zext-to-i32, align 2) still VMEM; the scalar sub-dword SMEM also needs
  align>=4-or-SMRD-imm-form AND no vector coalesce. (t16nc/a8w4 satisfy it via scattered scalar
  extloads with constant SMRD offsets.)
- **H5 "it is an LLVM-version effect."** REFUTED — identical selection on pins 56421f92 and
  62b7cf96 for every decisive case.

## CLAIM STATUS

**CLAIM (answered, irrefutable at ISel/asm level):** the truth table above. `s_load` selection is
gated by SIISelLowering.cpp:13148-13158 (uniform + invariant/noclobber + align>=4 +
global/const + scalarize-global) and by representability: dword-multiples coalesce to wide
`s_load_dwordx*`; sub-dword loads have a scalar-only SMEM form (`s_load_u8/u16`, one SGPR,
extload-to-i32), and NO vector SMEM form (SMInstructions.td:329-332 + 1044-1049), so a contiguous
sub-dword load that vectorizes to `<N x i16>`/`<N x i8>` is forced to VMEM even when
uniform+noclobber. That is the sub-dword ceiling.

**Counter-experiments that would refute (and their results):**
1. "Find a contiguous i16/i8 uniform+noclobber load that ISel puts in SMEM." -> i16c8/i8c8/t16 all
   VMEM; DAG shows `v8i16 ... invariant amdgpu-noclobber` still -> flat_load. Not refuted.
2. "Find a dword-granular uniform+noclobber+align>=4 load that ISel keeps in VMEM." -> i32c8/i64c4/
   t32/i32_noalias_store all wide/scalar SMEM. Not refuted.
3. "Show the sub-dword SMEM works from a vector instruction (s_load_u16x4)." -> no such def
   exists (Fact B grep NONE). Not refuted.
4. "Show align<4 native sub-dword still SMEM." -> one0_a2 VMEM; one0_a4 SMEM. Not refuted.
5. "Show divergent-address load still SMEM." -> i16/i32_divergent both VMEM. Not refuted.
6. "Show it's an LLVM-version artifact." -> two pins identical. Not refuted.

## Remaining unknowns (hand-offs, not blockers)

- **Native-subword scalar quirk (minor):** a *single* native i16 load (result used as i16) needs
  align>=4 to reach SMEM (one0_a2 VMEM), whereas scattered i16 loads whose results feed i32
  arithmetic reach SMEM at align 2 via the `S_LOAD_U16_IMM->i32` extload pattern (t16nc). The
  exact combiner condition distinguishing "widen aligned subword to dword" vs "keep scalar
  extload" was localized to the smrd_extload patterns + align check but not exhaustively mapped;
  it does not affect a8w4 (index is immediately `//N` => i32 use => scalar extload path).
- **Widening a8w4 to wide SMEM (S10/T3 territory):** the only route to *wide* SMEM for the u16
  index is dword granularity (load 2 adjacent u16 as one i32 + unpack, or change index dtype to
  i32 as moe_gfx1250 does -> `s_load_b512`). Verified in isolation (@i32c8/@t32 + moe's real
  `s_load_b512`); the combined a8w4 kernel change (drop %M + host-pad + i32-pack) is unimplemented
  (widen_sload README). This is a codegen-shape question, not a noalias-mechanism question.
- **buffer_load / s_buffer_load** as an alternative wide sub-dword path is S6/T3, not this cell.
