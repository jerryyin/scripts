# S7 — T4 (× F1/F3): does noalias unblock LoadStoreVectorizer / SLPVectorizer / SILoadStoreOptimizer widening?

## Construction (why this A/not-A isolates aliasing as the only variable)

Two adjacent loads from base `p` (offsets 0 and 4 bytes), separated by a `store` to
a *different* base `q`. LSV may merge `load p[0]` + `load p[1]` into one wide load
only if it can move one load across the intervening `store %q`, which is legal iff
`q` does NOT alias `p`. The only thing toggled between A and not-A is the `noalias`
param attr; body/offsets/alignment are byte-identical, so any width change is
attributable to aliasing information alone.

## HOW-facts (real IR / asm / MI lines — not paraphrased)

### Fact 1 — LoadStoreVectorizer (IR level), `opt -passes='load-store-vectorizer'`
`/tmp/s7/lsv_case.ll` → `/tmp/s7/lsv_out.ll`:

MAY-ALIAS (`@twoload_mayalias`, bare args) — NOT merged, stays two scalar loads:
```
  %a = load float, ptr addrspace(1) %p0, align 8
  store float %v, ptr addrspace(1) %q, align 4
  %b = load float, ptr addrspace(1) %p1, align 4
  store float %s, ptr addrspace(1) %q, align 4
```
NOALIAS (`@twoload_noalias`, `noalias` on p and q) — merged, one wide load hoisted:
```
  %0 = load <2 x float>, ptr addrspace(1) %p0, align 8
  store float %v, ptr addrspace(1) %q, align 4
  store float %s, ptr addrspace(1) %q, align 4
```
Identical decision on the second pin (62b7cf96): may-alias = 2 scalar loads,
noalias = 1 `load <2 x float>`.

### Fact 1b — store side is symmetric (`/tmp/s7/store_case.ll`)
Two stores to `p` separated by a load from `q`:
- may-alias: `store float ... %p0` + `store float ... %p1` (2 scalar stores).
- noalias:   `store <2 x float> %1, ptr addrspace(1) %p0, align 8` (1 wide store).

### Fact 2 — A/not-A kill-switch, necessary AND sufficient (`/tmp/s7/killswitch.ll`)
Four combos, count of `load <2 x float>` after LSV:
```
c_none  (bare,    bare)    : 0    <- may-alias, NOT vectorized
c_ponly (noalias, bare)    : 1    <- vectorized
c_qonly (bare,    noalias) : 1    <- vectorized
c_both  (noalias, noalias) : 1    <- vectorized
```
- NECESSARY: with neither annotated, no widening (`c_none = 0`).
- SUFFICIENT: `noalias` on EITHER base alone flips it (`c_ponly` / `c_qonly = 1`).
  BasicAA: `noalias %p` ⇒ p does not alias q; `noalias %q` ⇒ q does not alias p;
  either establishes `p !alias q`, which is all LSV/AA needs.

### Fact 3 — asm width change, `llc -O3` on the LSV-vectorized IR (`/tmp/s7/lsv_out.s`)
- may-alias: `s_load_b32 s4, s[0:1], 0x0` + `global_load_b32 v1, v0, s[0:1] offset:4`
  (two separate 32-bit loads, and one even falls to a per-lane global load).
- noalias:   `s_load_b64 s[0:1], s[0:1], 0x0`  (single 2-dword scalar load).

### Fact 4 — vectorization (IR) vs coalescing (MI) DISTINGUISHED
Feed the RAW, non-LSV'd scalar IR straight to `llc -O3 -amdgpu-load-store-vectorizer=0`
(IR-level LSV disabled inside llc), so only the backend (incl. SILoadStoreOptimizer)
can widen (`/tmp/s7/nolsv.s`):
- may-alias: still `s_load_b32` + `global_load_b32 offset:4` — NOT coalesced.
- noalias:   `s_load_b64` — widened.

MI before/after `-print-{before,after}=si-load-store-opt` proves WHICH pass does it:

NOALIAS (`/tmp/s7/na_only.ll`) — both data loads are selected as scalar noclobber
`S_LOAD_DWORD_IMM` (offset 0 and 4), THEN SILoadStoreOptimizer merges them:
```
Before si-load-store-opt:
  %8 = S_LOAD_DWORD_IMM %5, 0, 0 :: ("amdgpu-noclobber" load (s32) from %ir.p.load ...)
  %9 = S_LOAD_DWORD_IMM %5, 4, 0 :: ("amdgpu-noclobber" load (s32) from %ir.p1 ...)
After  si-load-store-opt:
  %13 = S_LOAD_DWORDX2_IMM %5, 0, 0 :: ("amdgpu-noclobber" load (s64) from %ir.p.load ...)
```
MAY-ALIAS (`/tmp/s7/ma_only.ll`) — p[0] is scalar noclobber but p[1] (after the
store to q) is a per-lane VECTOR global load; SILoadStoreOptimizer canNOT merge them:
```
Before si-load-store-opt:
  %9  = S_LOAD_DWORD_IMM        %6, 0, 0 :: ("amdgpu-noclobber" load (s32) from %ir.p.load ...)
  %11 = GLOBAL_LOAD_DWORD_SADDR %6, %5, 4, 0 :: (load (s32) from %ir.p1 ...)
After  si-load-store-opt:
  %9  = S_LOAD_DWORD_IMM        %6, 0, 0 :: (... unchanged ...)
  %11 = GLOBAL_LOAD_DWORD_SADDR %6, %5, 4, 0 :: (... unchanged, NOT merged ...)
  (only unrelated kernarg S_LOADs were merged into %16 = S_LOAD_DWORDX4)
```

Mechanism trace (the one non-obvious hop): `noalias %p` → FunctionAttrs/AA marks the
`p` loads `"amdgpu-noclobber"` (no store can clobber them) → BOTH become uniform
scalar `S_LOAD_DWORD` even the one after the `store %q` → they are now same-class
adjacent siblings → SILoadStoreOptimizer's scalar-load merge fires → `S_LOAD_DWORDX2`.
Under may-alias, the post-store load `p[1]` is provably clobberable by `store %q`, so
it is NOT noclobber, stays a vector `GLOBAL_LOAD_DWORD`, and is therefore ineligible
for the scalar merge. So SILoadStoreOptimizer is not "blocked by aliasing" per se — it
is unblocked *indirectly*: noalias changes the instruction selected (scalar vs vector +
noclobber), and only then are the two loads mergeable.

## Hypotheses + refutation trail

- **H1: "noalias is irrelevant to widening; -O3 widens anyway."** REFUTED — Fact 1/3/4:
  every may-alias variant stays two scalar loads at IR, asm, and MI level across both
  LLVM pins and with LSV on or off.
- **H2: "It's only IR-level LSV; the backend does nothing."** REFUTED — Fact 4: with
  IR LSV disabled in llc, noalias still yields `s_load_b64`, produced by
  SILoadStoreOptimizer merging two scalar `S_LOAD_DWORD` (MI before/after shown). Both
  levels participate; they are separable and both gated by aliasing info.
- **H3: "SILoadStoreOptimizer coalesces regardless of aliasing (pure address
  adjacency)."** REFUTED — Fact 4 may-alias MI: the two adjacent p-loads are NOT
  merged because one is scalar-noclobber and the other is a vector global load; the
  aliasing verdict changed the instruction class, blocking the merge.
- **H4: "SLPVectorizer is the pass at play."** REFUTED as scope — SLP vectorizes
  independent scalar *arithmetic* into vector ALU ops; the widening observed here is
  purely *memory* widening (LSV in IR, SILoadStoreOptimizer in MI). SLP is not the
  mechanism for these load/store cases. (Not exhaustively probed for a contrived
  SLP-memory case; see unknowns.)
- **H5: "It's a MachineLICM-patch / LLVM-version effect."** REFUTED — identical LSV
  decision on stock `56421f92` and `62b7cf96`; the patched install differs only by the
  orthogonal uniform-loop readfirstlane-hoist commit.

## CLAIM STATUS

**CLAIM (answered, irrefutable at IR + MI + asm level):**
`noalias` unblocks load/store WIDENING that may-alias blocks, at two distinct levels:
1. **Vectorization (IR):** LoadStoreVectorizer merges adjacent scalar loads/stores of
   one base into a single `load/store <N x T>` iff an intervening opposite-base access
   is proven non-aliasing. `noalias` supplies that proof.
2. **Coalescing (MI):** SILoadStoreOptimizer merges adjacent `S_LOAD_DWORD` →
   `S_LOAD_DWORDX2`, but only after `noalias`→`amdgpu-noclobber` has made BOTH loads
   uniform-scalar; under may-alias the post-store load stays a vector global load and
   is ineligible. So SILoadStoreOptimizer is unblocked *indirectly* by noalias (via
   instruction selection), not by feeding it aliasing metadata directly.
- **necessary/sufficient:** For the constructed widening, `noalias` is NECESSARY
  (Fact 2: `c_none = 0`) and SUFFICIENT (Fact 2: `c_ponly = c_qonly = 1`, noalias on
  either base alone). SLPVectorizer: NEITHER (wrong pass for these memory cases).

**Counter-experiment that would refute (and its result):**
- *"Show a may-alias variant that still widens to <2 x float> / s_load_b64."*
  → Result across 4 combos, 2 pins, LSV-on and LSV-off, IR+asm+MI: never. Not refuted.
- *"Show noalias fails to widen (so it's not sufficient)."* → Result: every noalias
  combo widens. Not refuted.
- *"Show SILoadStoreOptimizer coalesces the may-alias loads at MI level."* → Fact 4
  may-alias MI after-dump: the two p-loads are unchanged (different classes). Not refuted.
