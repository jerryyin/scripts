## HOW-facts (evidence)

### Fact 1 — the conversion mechanism (MLIR LLVM-dialect level), isolated
`triton-opt --convert-triton-amdgpu-to-llvm=gfx-arch=gfx1250` on a 2-ptr func where
only the first arg has `tt.noalias`:

```
llvm.func @contract(
  %arg0: !llvm.ptr<1> {llvm.noalias, tt.divisibility = 16 : i32, tt.noalias = 1 : i32, tt.pointee_type = i16},
  %arg1: !llvm.ptr<1> {tt.divisibility = 16 : i32, tt.pointee_type = f16},
  %arg2: !llvm.ptr<1>, %arg3: !llvm.ptr<1>) { llvm.return }
```
- `tt.noalias` arg → gains `llvm.noalias`. Plain arg → nothing.
- The AMD conversion attaches **only** `llvm.noalias` (contract) + Triton-internal
  metadata `tt.pointee_type` / carried `tt.divisibility`. It emits **no** `llvm.readonly`,
  `llvm.writeonly`, `llvm.align`, `llvm.dereferenceable`, `llvm.nocapture`, `llvm.noundef`.
  Notably `tt.divisibility = 16` is **not** turned into `llvm.align 16`.
- `%arg2/%arg3` = the global-scratch + profile pointers appended by `amendFuncOp`.

### Fact 2 — the real kernel today, PRE-opt LLVM IR (current tree tip, contract landed)
`define` line from `LLVM_IR_ENABLE_DUMP` (`/tmp/s1_preopt_dump.txt`):

```
define amdgpu_kernel void @_moe_gemm_a8w4_prefill(
  ptr addrspace(1) %0, i32 %1, ptr addrspace(1) %2, ... ,
  ptr addrspace(1) %14, i32 %15, i32 %16, i32 %17, i32 %18,
  ptr addrspace(1) noalias %19,          <-- GatherIndx
  ptr addrspace(1) %20, ptr addrspace(1) %21, ptr addrspace(1) %22,
  ptr addrspace(1) %23, i32 %24, i32 %25, float %26, float %27,
  ptr addrspace(1) %28, ptr addrspace(1) %29) #0 !dbg !4 {
```
- Ptr args in order: `%0 %2 %6 %9 %12 %13 %14  %19(noalias)  %20 %21 %22 %23  %28 %29`.
- **Exactly one** `noalias` on the whole define. **Zero** other param attrs
  (grep for readonly|writeonly|align N|dereferenceable|captures|noundef|nonnull|nocapture
  on the pre-opt define line → empty).
- Pre-opt `attributes #0` = only `amdgpu-cluster-dims`, `amdgpu-flat-work-group-size`,
  `denormal-fp-math-f32`, `uniform-work-group-size` — no aliasing hints.

### Fact 3 — arg %19 == GatherIndx (mapping proof)
Fresh TTGIR ptr-arg order: `Y, X, W, WMxScale, X_static_scale, Quant_static_scale,
B, GatherIndx, ExptHist, ExptOffs, ExptOffsSum, ExptData` (12 kernel ptrs). The
pre-opt define has 14 ptr args = those 12 + 2 appended scratch/profile. The **8th**
ptr in both is `GatherIndx` ↔ `noalias %19`. Every other named pointer (Y/X/W/
WMxScale/X_static_scale/Quant_static_scale/B/ExptHist/ExptOffs/ExptOffsSum/ExptData)
is a **bare** `ptr addrspace(1)`.

### Fact 4 — the frontend gate (source of `tt.noalias`) and its default
`python/triton/runtime/jit.py`:
- `KernelParam.__init__(..., noalias: bool = False)` — **default is False**.
- `_get_options...`: `if p.noalias and specialization[p.num][0] != "constexpr":
   attrs.setdefault((p.num,), []).append(["tt.noalias", 1])` — the attr is emitted
  **only** for a param explicitly listed in `noalias_args` **and** passed as a real
  (non-`None`/non-constexpr) pointer.
Fresh TTGIR confirms the frontend applied it:
`%GatherIndx: !tt.ptr<i16> {tt.divisibility = 16, tt.noalias = 1, tt.pointer_range = 32}`.

### Fact 5 — the single point of truth in the backend
`grep noalias` across `lib/`+`third_party/` conversion code: the **only** place a
*function-argument* `noalias` is set is
`lib/Conversion/TritonGPUToLLVM/Utility.cpp:2244` (`handlePointerContractArgs`,
called from `third_party/amd/lib/TritonAMDGPUToLLVM/FuncOpToLLVM.cpp:29`). All other
`NoAlias` hits (`addLocalLoadNoAliasScope`, `forceNoAlias`) are **LDS/shared-load
alias-scope metadata on load ops**, not param attributes — orthogonal to arg aliasing.
The generic `lib/Conversion/TritonGPUToLLVM/FuncOpToLLVM.cpp:60` `llvm.align` is set
**only** inside the `tt.nv_tma_desc` (NVIDIA TMA byval) path → never on AMD gfx1250.

### Fact 6 — family baseline (the "before any contract" state across AITer)
`grep -rl noalias_args ~/aiter/aiter/ops/triton/`: **exactly one file**,
`_gluon_kernels/gfx1250/moe/moe_op_gemm_a8w4.py`. attention=0, gemm=0, norm=0,
fusions=0. So across the AITer gfx1250 gluon tree, **all pointer args are bare
(may-alias) today except the single MoE GatherIndx**.

## Hypotheses + refutation trail

- **H1: "Triton emits noalias by default (graphics heritage)."** REFUTED — Fact 2
  (only 1 noalias, on the opt-in arg), Fact 4 (`noalias` defaults False), Fact 6
  (only one kernel opts in). Default is bare / may-alias.
- **H2: "Triton emits some other alias/access attr (readonly/align/deref/nocapture)
  by default that already helps."** REFUTED — Facts 1 & 2: the AMD path emits none of
  these on kernel args pre-opt; `tt.divisibility` is NOT lowered to `llvm.align`.
  (`readonly`/`captures(none)` seen in cached `.llir` are LLVM *post-opt inferences*,
  not emissions — and are exactly what T1 will show `noalias` unblocks.)
- **H3: "The cached `.llir` (no `noalias`) proves the contract doesn't apply."**
  REFUTED — those caches are stale/pre-contract (their TTGIR lacks `tt.noalias`)
  and post-opt; the fresh pre-opt dump on the tip has the `noalias` (Fact 2/3).

## CLAIM STATUS

**CLAIM (answered, irrefutable at compile/IR level):**
1. **What attrs today:** On gfx1250, Triton/AITer gluon kernel pointer args carry
   **no LLVM alias/access param attributes by default** — they are bare
   `ptr addrspace(1)`. The **only** exception in the whole AITer tree is
   `moe_op_gemm_a8w4.py`'s `GatherIndx`, which — via the opt-in `noalias_args`
   contract — carries exactly `noalias` (and nothing else: no readonly/align/
   dereferenceable/nocapture). `tt.divisibility`/`tt.pointer_range`/`tt.pointee_type`
   exist on the `tt.func`/`llvm.func` arg but are **Triton-internal MLIR metadata,
   not LLVM param attributes**, and `tt.divisibility` is specifically NOT lowered to
   `llvm.align` on AMD.
2. **Where set:** frontend gate `jit.py` (`noalias=False` default →
   `tt.noalias` attr only when opted in) → single backend sink
   `handlePointerContractArgs` (Utility.cpp:2233) invoked from AMD
   `FuncOpToLLVM.cpp` → `llvm.noalias`. No default-noalias code path exists.
3. **Default aliasing assumption + why:** Because Triton emits nothing, distinct
   kernel pointer args fall to **LLVM's default: they MAY alias** (conservative
   may-alias). This is the compute-language heritage — CUDA/HIP `__global__`
   pointer params are not `__restrict__` unless the author says so; Triton mirrors
   that opt-in model (`noalias_args`) rather than the graphics-API stance where
   distinct resource bindings are assumed non-overlapping (noalias by construction).
   The claim rests on *behavior* (Fact 4/5: opt-in only, single sink), not on a
   Triton doc — Triton ships **no** doc asserting a pointer-aliasing rule
   (`grep alias docs/` finds only polyhedral-related-work prose).

**Counter-experiment that would refute (and its result):**
- *"Compile a kernel with NO `noalias_args` and find a `noalias`/`restrict` on any
  ptr arg in pre-opt IR."* → Result: every other pointer in `_moe_gemm_a8w4_prefill`
  (all 11 non-GatherIndx kernel ptrs) is bare in the pre-opt define; and the isolated
  `triton-opt` probe shows the non-tagged arg gets nothing. Not refuted.
- *"Show the AMD conversion attaches align/readonly/etc. by default."* → Fact 1
  output shows it does not. Not refuted.

## Remaining unknowns (hand-offs, not blockers)

- **Post-opt fate of `noalias` (→ T1/S4):** the *post-opt* `.llir` shows the arg
  `noalias` **absorbed** and `readonly captures(none)` *inferred* instead — i.e.
  `noalias` is consumed by FunctionAttrs to derive `readonly`. That inference chain
  (noalias ⇒ readonly ⇒ invariant ⇒ LICM ⇒ s_load) is exactly S4/S5's job; S1 only
  establishes that the *emission* is `noalias`-only.
- Whether emitting `llvm.align`/`dereferenceable` from `tt.divisibility`/
  `tt.pointer_range` would independently help is a T2/T3/T7 question, not baseline.
- `pointer_range = 32` (32-bit offset promise) is present on all ptr args as MLIR
  metadata; its interaction with addressing is a T2/T3 concern.
